## 诊断要点
- 检查 Supabase 项目是否一致：应用使用的 `VITE_SUPABASE_URL`/`VITE_SUPABASE_ANON_KEY` 是否指向和你截图同一个项目。代码位置：utils/supabaseClient.ts:4-7。
- 核验查询时间范围：前端按当前视图计算区间再查询数据库，若区间外的记录不会显示。相关代码：components/App.tsx:107-119、utils/dateUtils.ts:115-142、utils/eventService.ts:55-57。
- 识别标签过滤：若右侧面板隐藏了某标签，对应事件会被过滤。相关代码：App.tsx:306-309。
- RLS 策略确保 `auth.uid() = user_id`，若不匹配无法读到数据。相关 SQL：supabase/migrations/001_events.sql:20-31。
- 时区边界：记录以 UTC 存储（+00:00），前端已转换为本地时间，但跨日事件在区间边界可能被排除。映射与本地化：utils/eventService.ts:14-36。

## 修改方案（最小改动）
1. 在加载与刷新时打印诊断信息（仅开发环境）：当前 `user_id`、视图的查询起止时间（ISO）、返回事件条数与首尾时间，便于快速定位是否为区间或项目不一致。
2. 优化事件查询条件：将“与区间有交集”的条件保留，但在日/周视图切换时增加一个兜底选项：
   - Day/Week：`gte('start_time', startDateISO).lt('start_time', nextEndISO)` 与现有条件并行尝试，取并集，避免跨日边界遗漏。
   - Month：维持现状。
3. 增加一个“忽略时间范围”调试开关（仅开发模式），直接按 `user_id` 拉取最近 N 天记录，用于确认环境变量/权限是否正确。
4. 标签可见性容错：当事件包含不存在于 `tags` 表的 `category` 字段时，自动将该分类加入 `visibleTags`，避免被错误隐藏（现有逻辑已做，补充在首次加载阶段强制合并）。

## 验证方式
- 本地：切到 Day/Week 视图，观察诊断日志与事件是否出现；在跨日事件（例如 23:30-01:00）场景下确认显示正常。
- SQL 侧：在 Supabase 控制台用你的 `user_id` 执行 `select * from events where user_id = '<uid>' order by start_time desc limit 50;` 验证确有数据。
- 端到端：登录同账号，核对首页右上角邮箱（App.tsx:588-592）与诊断日志中的 `user_id` 一致。
