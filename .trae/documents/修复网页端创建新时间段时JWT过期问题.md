## 问题分析
当创建新时间段时显示"JWT expired"错误，退出重新登录后暂时解决，但过一会儿又出现。这是因为Supabase JWT令牌有过期时间（默认1小时），而应用没有正确处理令牌过期的情况。

## 根本原因
1. Supabase JWT令牌过期后，API请求失败返回"JWT expired"错误
2. 应用没有捕获并处理这个特定错误
3. 缺少令牌自动刷新机制的错误处理
4. API调用前没有验证会话状态

## 修复方案
### 核心修复（不影响数据）
1. **在eventService.ts中添加JWT过期错误处理**：
   - 在createEvents、updateEvent、deleteEvent等函数中添加错误捕获
   - 识别"JWT expired"错误，并触发会话刷新
   - 确保API调用失败时不会导致数据丢失

2. **改进App.tsx中的会话管理**：
   - 增强onAuthStateChange监听器，处理令牌过期情况
   - 添加会话验证机制，确保API调用前会话有效
   - 添加全局错误处理，统一处理JWT过期错误

3. **在supabaseClient.ts中配置自动刷新**：
   - 确保Supabase客户端配置了正确的令牌刷新选项
   - 添加刷新失败的处理逻辑

4. **添加用户提示机制**：
   - 当令牌过期无法自动刷新时，提示用户重新登录
   - 提供清晰的错误信息，避免用户困惑

### 数据安全保障措施
1. **数据完整性确认**：
   - 在修复前运行数据检查，确认当前所有时间段数据完整
   - 记录当前数据状态，作为修复后的对比基准

2. **数据备份机制**：
   - 添加本地数据备份功能，在API调用前自动备份当前事件数据
   - 实现数据恢复功能，以防万一出现数据问题

3. **安全的API调用**：
   - 确保所有数据操作都使用事务处理，要么完全成功，要么完全失败
   - 添加操作日志，记录所有数据修改，便于追踪和恢复

4. **数据验证机制**：
   - 修复后验证数据完整性，确保所有之前的时间段记录都存在
   - 提供数据一致性检查工具，帮助用户确认数据完整

## 实施步骤
1. **数据完整性检查**：运行数据检查脚本，确认当前所有时间段数据完整
2. **修改eventService.ts**：添加JWT过期错误处理，确保API调用安全
3. **修改App.tsx**：增强会话管理和错误处理，不涉及数据操作
4. **检查supabaseClient.ts**：配置正确的令牌刷新选项
5. **添加数据备份机制**：实现本地数据备份和恢复功能
6. **测试修复效果**：验证创建事件时不再出现JWT过期错误
7. **数据完整性验证**：修复后再次检查数据，确保所有之前的时间段记录都存在

## 预期效果
- 创建新时间段时不再显示"JWT expired"错误
- 令牌过期时自动刷新，无需用户手动重新登录
- 所有之前的时间段数据记录保持完整
- 增加了数据备份和恢复机制，提高数据安全性
- 提供清晰的错误信息和用户指导

## 数据安全承诺
- 所有修复操作都不会直接修改或删除数据库中的时间段数据
- 修复过程中添加的数据备份机制只会增加数据安全性，不会影响现有数据
- 修复后会进行数据完整性验证，确保所有之前的时间段记录都存在
- 提供数据恢复方案，以防万一出现数据问题
