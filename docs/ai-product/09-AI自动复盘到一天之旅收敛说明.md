# AI 自动复盘到「一天之旅」收敛说明

## 1. 文件定位

本文件用于解释 `docs/ai-product/` 和 `docs/ai-day-journey/` 之间的关系。

`docs/ai-product/` 是 AI 功能探索和上位规划区，记录的是「为什么要做 AI 复盘」「第一版 AI 功能应该选什么」「如何从 AI 产品经理视角定义 Workflow、Eval、Badcase」。

`docs/ai-day-journey/` 是最终落地功能区，记录的是「一天之旅」从人工范本、系统提示词、接口、前端看板、Golden Set、Consistency Eval 到 Supabase 保存链路的完整落地过程。

## 2. 原始方向：AI 自动日复盘

早期规划中，第一版 AI 功能被定义为「AI 自动日复盘」。

核心思路是：

- 输入：当天 events、tags、时间分布、标签统计、事件数量等。
- 输出：今日概览、时间分配、关键洞察、风险提醒、明日建议、数据限制。
- 技术形态：LLM + Workflow。
- 非目标：不做 Agent、不做 RAG、不做 Chrome 插件、不做订阅系统。

这一路径的价值是帮用户从“记录时间”走到“理解时间”。

## 3. 最终收敛：一天之旅

实际落地过程中，AI 自动日复盘没有继续停留在通用 JSON 总结形态，而是收敛成更具体、更可评估、更有作品集辨识度的「一天之旅」。

「一天之旅」保留了 AI 自动日复盘的核心目标，但改变了输出形态：

| 维度 | 早期 AI 自动日复盘 | 最终一天之旅 |
|---|---|---|
| 输出形态 | JSON / 结构化复盘 | Markdown 文档 |
| 内容风格 | 概览、洞察、风险、建议 | 柳比歇夫式时间账本 |
| 质量标准 | 事实准确、建议可执行 | Golden Set、段落组织、关键词、事实边界 |
| 评测方式 | Eval Rubric + Badcase | Golden Reference + Consistency Eval + Badcase |
| 用户价值 | 看懂今天时间分布 | 把一天变成可阅读、可保存、可回查的复盘资产 |
| 数据沉淀 | 早期规划未强制保存 | `ai_day_journeys` 按 `user_id + date` 保存最新版 |

## 4. 为什么这个收敛是合理的

选择「一天之旅」不是偏离 AI 自动复盘，而是把一个泛功能收敛成了更清晰的 MVP。

原因包括：

1. **输出更可控**：固定 Markdown 格式比开放式总结更容易做验收。
2. **质量更可评估**：6 月 1 日到 6 月 4 日 Golden Set 可以直接作为参考标准。
3. **Badcase 更具体**：时间点失真、关键词遗漏、段落过碎、事实污染都能被明确记录。
4. **作品集表达更完整**：能讲清楚 Prompt、接口、Eval、Badcase、Consistency Eval、Supabase 保存。
5. **更贴近用户真实场景**：用户不是只需要一句总结，而是需要一份可以回看、复制、保存和沉淀的日复盘内容。

## 6. 面试表达方式

可以这样讲：

> 我最开始没有直接做一个炫技型 Agent，而是先从当前产品数据出发，做了非 AI 版 PRD 和 AI 功能探索。早期方向叫 AI 自动日复盘，目标是把 events 和 tags 转成可读复盘。后来在落地过程中，我发现泛化的“总结 + 建议”不够有辨识度，也不好评估，所以把功能收敛成「一天之旅」：固定 Markdown 输出、固定早中晚结构、基于 Golden Set 做 Eval，并用 Badcase 修复时间点失真、事实遗漏、段落过碎和 Golden 事实污染。最终它形成了一个 LLM + Workflow 功能：读取当天时间记录，调用模型生成一天之旅，前端渲染展示，并支持按日期保存到 Supabase。

这个表达能体现：

- 不是为了接 API 而接 API。
- 有产品选题判断。
- 有 MVP 收敛能力。
- 有 Prompt 和输入构造意识。
- 有 Eval / Badcase / Consistency Eval 思维。
- 有数据资产和保存闭环意识。

## 7. 当前同步结论

`docs/ai-product/` 中早期 AI 自动复盘文档可以保留，不需要全部改写成一天之旅。它们的价值是记录选题和上位规划。

`docs/ai-day-journey/` 是最终功能证据中心，后续简历、作品集和面试复盘应以该目录为主。
