# AI 功能探索与选择记录

## 1. 候选 AI 功能方向

本记录用于衔接 `00-时间管理项目当前现状-非AI版PRD-V0.md` 与 `01-AI自动复盘-PRD-V0.md`，回答一个更前置的问题：在当前个人时间管理 Web/PWA 已有数据、6.8 前交付节奏、AI 产品经理求职证据包目标下，第一版 AI 功能应该优先选择什么。

当前项目已有基础包括：日历时间块、事件 CRUD、标签分类、计时器 / 秒表、历史记录、Supabase 同步、ICS / JSON 导入导出。当前缺口包括：没有真实 LLM 调用、AI 自动复盘、AI 结果存储、Eval、Badcase 和用户反馈闭环。

候选方向如下：

1. AI 自动日复盘：基于当天 events 和 tags，生成今日概览、时间分布、关键洞察、风险提醒和明日建议。
2. AI 自动周复盘：基于近 7 天 events 和 tags，生成周维度投入结构、趋势观察和下周建议。
3. AI 标签时间分布分析：按标签统计时间占比，并用自然语言解释不同标签投入情况。
4. AI 异常提醒：识别休息过少、晚间任务过多、碎片事件过多、单类标签占比异常等情况。
5. AI 明日行动建议：基于当天记录，给出 2-4 条可执行的明日调整建议。
6. AI 时间记录质量检查：检查当天记录是否过少、标签缺失、时长异常、标题过短或重复记录。
7. AI 日记 / 时间记录联合复盘：结合用户日记、目标或备注，与时间记录一起做更深层复盘。
8. AI 作品集报告生成：把用户一段时间的学习、工作和时间投入自动整理成求职作品集报告。

## 3. 推荐结论

第一版最应该做的 AI 功能是：**AI 自动日复盘**。

推荐原因：

- 场景足够收敛：只处理当天 events 和 tags，不需要新增日记、目标、偏好、浏览器行为或订阅系统。
- 数据边界清楚：输入来自当前项目已有的事件标题、起止时间、标签、日期和规则统计，适合先做 LLM + Workflow。
- 6.8 前可交付：日复盘能快速形成 PRD、Workflow、Eval、Badcase、Prompt、Demo 截图或录屏，适合作为最低投递证据包。
- AI PM 作品集表达更完整：它不只是“接一个模型 API”，而是能讲清楚为什么要先规则聚合、为什么不用 RAG / Agent、如何评估输出、如何沉淀 Badcase。
- 评估闭环容易建立：事实一致性、结构完整性、建议可执行性、克制程度、兜底质量都可以转成 Eval 维度。
- 风险可控：AI 只生成复盘和建议，不自动修改用户日程，不触碰高风险执行权限。

第一版可以合并两个辅助能力，但不单独作为主功能：

- **AI 时间记录质量检查**：作为生成复盘前的输入校验，识别无记录、记录太少、标签缺失、时长异常等情况。
- **AI 明日行动建议**：作为日复盘输出的一部分，限制为 2-4 条轻量、具体、基于当天数据的建议。

暂不作为第一版主功能：

- **AI 自动周复盘**：适合 V1 或 Demo 扩展，但第一版会增加跨天趋势和数据完整性问题。
- **AI 标签时间分布分析**：适合作为日复盘中的 `time_distribution` 模块，独立成功能作品集故事偏薄。
- **AI 异常提醒**：可以作为日复盘中的风险提醒，但没有目标和偏好时不宜独立做强判断。
- **AI 日记 / 时间记录联合复盘**：需要新增日记或目标数据，后续可以作为 RAG 方向。
- **AI 作品集报告生成**：适合作为项目包装材料，不适合作为第一版产品主功能。

因此，V0 的产品策略应是：**用 AI 自动日复盘作为主功能，用时间记录质量检查保证输入可靠，用明日行动建议补齐行动闭环；不做复杂 RAG、完整 Agent、Chrome 插件和订阅系统。**

## 4. 第一版 AI 功能定义卡

| 项目 | 定义 |
| --- | --- |
| 功能名称 | AI 自动日复盘 V0 |
| 一句话定义 | 基于当天 events 和 tags，生成结构化时间复盘、风险提醒和明日行动建议。 |
| 目标用户 | 已有时间记录习惯、正在做学习 / 求职 / 个人效率管理的用户。 |
| 核心场景 | 用户晚上查看当天历史记录或日历面板时，点击生成 AI 日复盘，快速理解今天时间花在哪里、哪些地方值得保留、明天可以调整什么。 |
| 输入数据 | 当天事件、标签、事件标题、开始时间、结束时间、计算时长、标签聚合、事件数量、最长事件、碎片事件、异常数据提示。 |
| 输出内容 | 今日概览、时间分布、关键洞察、风险提醒、明日建议、数据限制。 |
| 技术形态 | LLM + Workflow。系统先做确定性清洗和聚合，再把结构化输入交给 LLM；V0 不做 RAG，不做 Agent。 |
| 不做事项 | 不自动修改日程；不判断目标达成；不接 Chrome 插件；不做订阅；不建立长期记忆；不新增数据库表作为 V0 必要条件。 |
| Eval 重点 | 事实一致性、结构完整性、洞察具体性、建议可执行性、克制程度、兜底质量。 |
| Badcase 重点 | 编造事实、过度推断、建议空泛、误读标签 / 时长、输出结构漂移、数据不足仍强行总结。 |
| Demo 交付物 | PRD、Workflow、Eval 表、Badcase 表、Prompt 版本记录、页面演示截图 / 录屏。 |
| 成功标准 | 能基于真实或样例 events / tags 生成可读、具体、克制的日复盘；至少覆盖正常记录、无记录、记录过少、标签缺失、碎片化记录等样本；能把失败输出沉淀到 Badcase。 |
| 作品集表达重点 | 展示 AI PM 的选题判断、范围控制、Workflow 设计、模型输出评估、Badcase 迭代意识，而不是展示复杂工程实现。 |
