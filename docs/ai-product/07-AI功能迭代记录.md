# AI 功能迭代记录

本文件记录“AI 自动复盘功能”的产品迭代总过程。它不是开发日志，也不是 Prompt 日志，而是用于串联每一轮需求调整、代码修改、Prompt 修改、输出结构修改、测试、验收和修复。

每次围绕 AI 自动复盘功能进行迭代时，都应在文件末尾追加一条记录，不覆盖旧记录。

## 迭代记录 2026-06-05 第 1 轮

### 1. 本轮迭代目标

把“AI 自动日复盘”从文档方案推进到可演示的最小功能：在日历右侧日程面板中增加 AI 复盘入口，打开结构化复盘弹窗，并建立本地 LLM 调用链。第一版目标是支持作品集 Demo，而不是完整上线版本。

### 2. 本轮修改内容

- UI 入口：在右侧日程面板工具按钮区，将 AI 复盘入口放在“筛选标签”按钮右侧。
- 按钮形态：入口只显示 AI 图标，不显示按钮文字，避免挤占日程面板空间。
- 复盘窗口：新增居中结构化复盘弹窗，包含数据依据、今日概览、时间分布、关键观察、风险提醒、明日建议和反馈按钮。
- 数据聚合：前端基于当前面板 `events` 和 `tags` 计算总时长、标签分布、主要投入方向、碎片记录和晚间记录。
- 失败态：当记录少于 2 条或总记录时长过短时，展示“记录不够生成可靠复盘”提示。
- LLM 调用：新增本地 Vite dev server API `/api/ai/daily-review`，服务端读取 `GLM_API_KEY` / `DEEPSEEK_API_KEY`，前端不直接接触 API Key。
- 模型优先级：优先调用 GLM；GLM 失败后再尝试 DeepSeek。
- 输出结构：服务端 Prompt 要求 LLM 输出 JSON，字段包含 `summary`、`time_distribution`、`insights`、`risks`、`actions`、`caveats`。
- 兜底逻辑：真实 AI 调用失败时，弹窗展示本地 Demo 复盘，并提示真实 AI 暂时不可用。
- 反馈闭环：弹窗底部提供“有帮助 / 不准确 / 太空泛 / 建议不可执行”四类反馈按钮，目前仅在前端展示“已记录反馈”，尚未入库。

### 4. 当前 AI 功能状态

当前 AI 自动日复盘已经具备可演示入口和弹窗。功能可以读取当前右侧面板传入的 `events` 和 `tags`，可以在前端生成本地结构化 Demo 复盘；在本地开发环境中，如果 `.env.local` 配置了 `GLM_API_KEY` 或 `DEEPSEEK_API_KEY`，会通过 `/api/ai/daily-review` 尝试生成真实结构化复盘。

当前实现仍属于本地 Demo 阶段：LLM 调用依赖 Vite dev server middleware，不是生产可部署 API；AI 结果和用户反馈尚未保存到数据库；Eval 和 Badcase 仍需后续补真实样本。

- 重启 `npm run dev -- --host 127.0.0.1`，确认本地开发服务器加载新的 Vite middleware。
- 未完成真实 LLM 输出人工验收；本轮主要验证代码构建和本地兜底链路。

### 6. 测试结果

- 未完成真实测试：尚未记录 GLM / DeepSeek 实际返回内容是否稳定符合 JSON 结构。

### 7. 发现的问题

- 当前 Vite middleware 只适合本地 Demo，后续上线需要迁移到 Supabase Edge Function、Vercel Function 或其他服务端代理。
- 真实 LLM 输出尚未经过 Eval 样本验证，可能出现 JSON 格式漂移、建议空泛、误读标签或过度推断。
- 当前反馈按钮没有持久化，只能作为 Demo 中的闭环表达。

### 9. Prompt 变化

本轮将调用 Prompt 从文档中的 Markdown 六段式方向，推进为实际接口中的 JSON 输出约束。Prompt 明确要求：

- 只基于输入 `events`、`tags` 和 `stats` 输出。
- 不编造目标、情绪、性格、完成状态或未提供事实。
- 只输出 JSON。
- 固定输出 `summary`、`time_distribution`、`insights`、`risks`、`actions`、`caveats`。

已同步记录到 `docs/ai-product/05-Prompt版本记录.md`。
