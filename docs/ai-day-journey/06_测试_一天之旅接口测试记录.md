# 一天之旅接口测试记录

## 0. 文件定位

本文件用于记录 `/api/ai/day-journey` 接口的本地测试过程、测试 payload、返回结果、基础结构检查和 Badcase。

## 1. 2026 年 6 月 4 日真实样例测试

### 1.1 测试目标

验证接口是否可以基于真实日程 / 时间记录生成「一天之旅」Markdown。

本轮只测试接口，不接入前端看板，不修改 `ReviewBoard.tsx`，不修改 Supabase。

### 1.2 测试数据

测试 payload：

```text
docs/ai-day-journey/test-payloads/2026-06-04-day-journey-payload.json
```

数据来源：

```text
docs/ai-day-journey/examples/2026年6月4日时间记录与一天之旅.md
```

payload 内容：

- 日期：`2026-06-04`
- 时区：`Asia/Shanghai`
- 事件数量：19 条
- 标签数量：6 个
- 时间处理：原始记录为 GMT+0，测试 payload 转换为 `+08:00` 北京时间。
- 字段保留：每条事件保留 `title`、`start`、`end`、`tagId`、`tagName`、`category`、`description`。

### 1.3 测试方式

启动本地开发服务后，通过 POST 请求调用：

```text
POST http://127.0.0.1:4177/api/ai/day-journey
```

本轮没有修改前端，也没有通过页面按钮触发。

### 1.4 空 events 兜底测试

请求摘要：

- 接口：`POST http://127.0.0.1:4177/api/ai/day-journey`
- 请求方式：POST
- 请求内容：只传入日期和空 events。
- 测试目的：确认空记录不会触发 LLM，也不会编造一天之旅。

原始 request body：

```json
{
  "date": "2026-06-04",
  "events": []
}
```

原始 response：

```json
{
  "success": false,
  "error": {
    "code": "NO_EVENTS",
    "message": "当前日期暂无可生成的一天之旅记录"
  }
}
```

结论：通过。空记录不会触发 LLM，不会编造一天之旅。

### 1.5.1 成功 response 字段解释

| 字段 | 本次值 | 含义 |
|---|---|---|
| `success` | `true` | 表示接口调用成功，并且已经拿到可返回给前端的结果。它只说明接口链路成功，不代表生成质量一定完全达标。 |
| `provider` | `glm` | 表示本次实际由哪个模型服务商生成结果。本接口优先尝试 GLM，失败后才会尝试 DeepSeek。 |
| `model` | `glm-4-flash` | 表示本次实际使用的具体模型名称。provider 是服务商，model 是具体型号。 |
| `promptVersion` | `day-journey-system-prompt-v1` | 表示本次生成使用的是哪一版系统提示词，方便后续复盘、Eval、Badcase 对比和作品集说明。 |
| `warnings` | `[]` | 表示接口基础结构检查没有发现缺失项。本次为空数组，说明日期、关键词、早上、下午、晚上、一天回顾等基础结构都存在。 |
| `markdown` | 约 1393 字符 | 模型生成的「一天之旅」正文，前端后续会把它展示在看板里，也可以复制或保存为 `.md`。 |

- 终端日志：本轮主要依据接口返回和 CodeX 执行结果，不依赖浏览器页面，也没有通过前端按钮触发。

### 1.7 输出质量观察

基础接口能力通过：接口可以接收真实 payload、调用 LLM、返回 Markdown，并通过最小结构检查。

但输出质量还没有完全贴近 `03_Golden_一天之旅最终范本.md`，存在以下 Badcase：

1. 时间格式不完全符合最终范本。
   - 返回内容使用了 `07:15 - 08:00：...` 这种时间范围格式。
   - 最终范本要求更接近 `7:15，...`，不堆叠完整起止范围。
   - 还出现了前导 0，例如 `07:15`，不符合最终范本中的 `7:15` 风格。

2. 早中晚分段不完全贴合最终范本。
   - 返回内容把 `11:17 - 11:36` 和 `11:36 - 12:14` 放进了下午。
   - 最高优先级最终范本中，这些内容仍归入早上段落，下午从 `12:14` 开始。

3. 一天回顾过短。
   - 返回的一天回顾三段只有高度压缩的一句话。
   - 与最终范本相比，遗漏了 yyw 聊天、贝索斯视频、Telegram 访谈、元思考、Canva 图标 Logo、午休后和许倩聊天、和学姐沟通、贵贵视频、晚饭散步洗澡、天物空间买东方树叶等关键事实。

4. 晚上阶段归纳有事实压缩遗漏。
   - 返回的晚上 `◆` 阶段归纳没有明确覆盖非 AI 版 PRD V0 产出、VS Code、GitHub、Git、终端和命令行基础学习等关键结果。

5. 没有发现明显虚构内容。
   - 本轮输出主要问题是格式和事实覆盖不足，不是编造事实。

### 1.8 本轮结论

接口 V0 技术链路通过：

- 可以接收真实 date / events / tags payload。
- 可以触发 LLM。
- 可以返回 `success: true`。
- 可以返回 Markdown。
- 可以带回 `promptVersion`、`provider` 和 `model`。
- 可以通过基础结构检查。

但生成质量还未达到最终范本标准。下一步接前端之前，建议先做一次 Prompt / 输入格式 / 校验策略优化，让输出更接近 2026 年 6 月 4 日最终范本。

## 2. 待跟进问题

| 问题 | 建议处理 |
|---|---|
| 时间格式使用范围和前导 0 | 在系统提示词或用户输入中进一步强调只用关键开始时间，如 `7:15，...` |
| 分段边界偏差 | 明确 6 月 4 日最终范本中上午 / 下午 / 晚上的分段参考，或在接口侧传入目标分段规则 |
| 一天回顾过短 | 强化“一天回顾也不能遗漏关键事实”，必要时增加字数 / 覆盖规则 |
| 基础校验过松 | 后续可加入时间格式、`◆` 数量、一天回顾三段长度、关键事实覆盖等更细校验 |

## 3. 第 4.6 步：V1.1 质量修复后重新测试

### 3.1 本轮修复内容

本轮基于第 4.5 步真实样例测试 Badcase，进行了以下修复：

- 将 `一天之旅系统提示词.md` 升级为 V1.1。
- 强化时间格式：不要输出 `07:15 - 08:00`，应输出 `7:15，...`。
- 强化分段规则：6 月 4 日样例中，`7:15` 到 `12:14` 前归入早上，`12:14` 后进入下午，`18:15` 后进入晚上。
- 强化事实保留：人名、地点、工具、项目、购买行为、交流对象、学习内容、生活事件和中途插入事项不能因为压缩而遗漏。
- 强化一天回顾：必须使用 ☀️ / 🌇 / 🌃 三段第一人称，并覆盖该时段主要事实。
- 优化接口传给模型的输入：除了 JSON，也传入 `events_readable_for_generation`，明确每条事件的本地时间、建议时段、分类符号、标题和原始描述。
- 增加质量 warnings：检查时间范围、前导 0、上午事件误入下午、一天回顾过短、`◆` 数量不足、结尾疑似截断。
- 增加模型输出上限，避免生成结果在末尾被截断。
- 修正 6 月 4 日分段边界：从 `12:14` 这一条开始进入下午，从 `18:15` 这一条开始进入晚上。
