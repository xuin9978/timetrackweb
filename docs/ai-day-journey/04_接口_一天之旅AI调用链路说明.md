# 一天之旅 AI 调用链路说明

## 0. 文件定位

本文件用于解释「一天之旅」AI 功能从用户点击按钮，到模型生成内容，再到前端展示结果的完整链路。

目标是帮助我理解一个 AI 功能如何从 0 到 1 跑通。

## 1. 当前目标链路

```text
用户点击「生成一天之旅」
→ 前端读取当前日期
→ 前端收集当天 events 和 tags
→ 前端调用 /api/ai/day-journey
→ 后端接口接收当天原始时间记录
→ 后端读取「一天之旅系统提示词稳定版 V1.6」
→ 后端调用 GLM / DeepSeek / OpenAI-compatible API
→ 模型生成一天之旅内容
→ 后端返回 Markdown
→ 前端渲染到「一天之旅」看板
→ 用户保存或重新生成
```

## 2. 当前已有链路

### 2.1 当前 AI 日复盘调用链路

```text

→ `components/EventPanel.tsx` 打开 `AIDailyReviewModal`
→ `components/AIDailyReviewModal.tsx` 接收当前 events、comparisonEvents、referenceDate、tags

→ 用户点击弹窗里的生成按钮
→ 前端通过 fetch 请求 `/api/ai/daily-review`
→ `vite.config.ts` 中的 Vite dev server middleware 接收请求
→ 后端拼接 AI 日复盘系统提示词和用户输入数据
→ 后端优先调用 GLM，失败后尝试 DeepSeek
→ 模型返回 JSON
→ 前端 normalize 成 `RemoteReview`
→ 弹窗展示 AI 摘要、洞察、建议和数据限制
```

## 3. 一天之旅目标链路

当前已完成 Prompt 层稳定化。当前 V0 接口会读取 `docs/ai-day-journey/一天之旅系统提示词.md`，并在 response 中返回 `promptVersion`。截至 V1.6，系统提示词重点从“生成格式可用”进一步收束为“事实边界稳定、同日期多次生成不漂移”。

当前已完成接口契约设计，并已实现 V0 接口：

```text
POST /api/ai/day-journey
```

### 3.0 当前实际调用链路 V0

```text
前端将 date、timezone、events、tags、options 发送到 /api/ai/day-journey
→ `vite.config.ts` 中的 Vite dev server middleware 接收 POST 请求
→ 接口校验 date 和 events
→ events 为空时直接返回 NO_EVENTS，不调用模型
→ 接口读取 `docs/ai-day-journey/一天之旅系统提示词.md`
→ 接口整理 events，兼容 start/end 和 startTime/endTime 字段
→ 接口同时构造 `events_json` 和可读事件清单 `events_readable_for_generation`
→ 接口调用 OpenAI-compatible 模型
→ 优先尝试 GLM，失败后尝试 DeepSeek
→ 模型返回 Markdown
→ 接口检查 Markdown 是否包含日期、关键词、早上、下午、晚上、一天回顾等必要结构
→ 接口补充质量检查：时间范围、前导 0、分段误放、一天回顾过短、结尾疑似截断
→ 接口返回 Markdown 和 warnings
```

当前 V0 复用了现有 AI 日复盘的模型调用方式：

- 继续读取环境变量中的 `GLM_API_KEY` 和 `DEEPSEEK_API_KEY`。
- 继续使用 OpenAI-compatible Chat Completions 请求格式。
- 继续采用 GLM 优先、DeepSeek 兜底的 provider fallback。
- 差异是：AI 日复盘要求模型返回 JSON；一天之旅要求模型返回 Markdown。
- 当前一天之旅 promptVersion 已升级为 `day-journey-system-prompt-v1.1`。
- 当前一天之旅接口会设置更高的模型输出上限，减少 Markdown 末尾被截断的概率。

### 3.5 6 月 1 日 / 6 月 2 日泛化测试链路

第 5.8 步中，6 月 1 日和 6 月 2 日用于验证泛化能力。

本轮调用链路：

```text
Golden md 中的原始日历记录
→ 手动整理为 test payload
→ 每条事件加入 displayStartTime / displayEndTime
→ POST /api/ai/day-journey
→ 系统提示词 V1.2 + 当前事件输入
→ 不注入 6 月 1 日 / 6 月 2 日 Golden Reference
→ GLM 返回 Markdown
→ 接口返回 warnings
→ 与 Golden md 做人工对比
```

本轮刻意没有把 6 月 1 日和 6 月 2 日加入 `DAY_JOURNEY_GOLDEN_REFERENCES`，目的是观察当前 Prompt 和输入构造在未直接参考答案时的泛化表现。

测试结果：

| 日期 | 接口结果 | warnings | Golden 对比 |
|---|---|---|---|
| 2026-06-01 | 成功返回 Markdown | 一天回顾偏短 | 未通过 |
| 2026-06-02 | 成功返回 Markdown | 缺少 `### 一天回顾` | 未通过 |

暴露出的链路问题：

- `displayStartTime` 已能防止时间点漂移。
- 但当前接口只做 warnings，不会在必要结构缺失时自动重试。
- 未注入 Golden Reference 的日期更容易出现关键词过泛、阶段归纳过粗和一天回顾缺失。
- 后续 V1.3 可考虑在接口层对缺少 `### 一天回顾` 的结果做一次重试，或在输入中加入更明确的输出结构检查提示。

V1.2 关键变化：

- `displayStartTime` 是正文时间点的唯一来源。
- 模型不得自行推算、平移、四舍五入或改写开始时间。
- Golden Set 是 Eval 和 few-shot 参考标准。
- 生成结果不要求逐字复制 Golden md，但必须对齐结构、事实覆盖、分段、关键词粒度、阶段归纳粒度和一天回顾风格。

### 3.7 轻量操作链路 V0

第 6 步后，「一天之旅」前端看板新增复制、下载和重新生成能力。

当前用户链路：

```text
用户点击右侧「一天之旅」图标
→ App 打开 ReviewBoard 并调用 generateDayJourney
→ POST /api/ai/day-journey
→ 接口返回 markdown
→ ReviewBoard 展示 markdown
→ 用户可以复制 Markdown / 下载 .md / 重新生成
```

操作说明：

- 复制 Markdown：`ReviewBoard` 将当前 `markdown` 写入剪贴板，并显示「已复制」或「复制失败，请手动复制」。
- 下载 `.md`：`ReviewBoard` 在浏览器本地生成 Markdown 文件，文件名使用当前选中日期，例如 `2026-06-04-一天之旅.md`。
- 重新生成：`ReviewBoard` 调用从 `App.tsx` 传入的 `generateDayJourney`，覆盖当前看板内容，不保存旧版本。
- 当时仍不保存到 Supabase，也没有历史版本。后续第 7 步已补充按日期保存到 Supabase 的链路，见本文 `3.10 Supabase 保存链路 V0`。

### 3.8 Markdown 展示层链路

第 6.1 步后，`ReviewBoard` 对 AI 返回的 Markdown 做展示层渲染。

当前链路：

```text
/api/ai/day-journey 返回原始 markdown
→ App.tsx 保存原始 markdown
→ ReviewBoard 阅读区渲染为可读视图
→ 复制 Markdown 使用原始 markdown
→ 下载 .md 使用原始 markdown
```

设计原则：

- 后端仍返回 Markdown，不改成纯文本。
- 展示层隐藏 `##`、`###`、`**` 等原始 Markdown 符号。
- 复制和下载保留原始 Markdown，方便后续保存、下载和作品集整理。
- 当前没有引入 Markdown 依赖，只在 `ReviewBoard` 内做轻量渲染。

### 3.9 V1.4 段落缩进链路

第 6.2 步后，「一天之旅」输出新增正文段落首行缩进规范。

当前链路：

```text
系统提示词 V1.4 要求正文段落以两个中文全角空格开头
→ /api/ai/day-journey 使用 day-journey-system-prompt-v1.4
→ 接口生成 Markdown 后检查正文段落缩进
→ 原始 markdown 保留 `　　`
→ ReviewBoard 展示层保留空白并渲染可读视图
→ 复制 / 下载继续使用原始 markdown
```

缩进规则：

- 正文段落、关键词段落、`◆` 阶段归纳、一天回顾三段必须以 `　　` 开头。
- `## 日期标题` 和 `###` 时段标题不缩进。
- 如果模型返回的正文行缺少 `　　`，接口会在 `warnings` 中提示。

### 3.10 Supabase 保存链路 V0

第 7 步后，「一天之旅」支持按用户和日期保存到 Supabase。

当前保存链路：

```text
用户选择日期
→ App 计算 selectedDateKey，格式为 YYYY-MM-DD
→ App 查询 ai_day_journeys 中 user_id + date 对应记录
→ 有记录则显示保存内容
→ 无记录则显示空状态
→ 用户点击生成一天之旅
→ /api/ai/day-journey 返回 markdown
→ 页面显示新生成内容，并标记为未保存
→ 用户点击保存
→ utils/dayJourneyService.ts upsert 到 Supabase
→ 保存成功后标记为已保存到当前日期
```

关键约束：

- 生成、保存、读取都使用同一个 `selectedDateKey`。
- `date` 字段不从 UTC / ISO 重新推导，避免保存到前一天或后一天。
- 同一用户同一天只保留一份最新版，重复保存使用 upsert 更新。
- 重新生成只更新页面，不自动保存。
- 切换日期时会先清空当前内容，再读取新日期保存内容。
- 旧日期异步请求返回时，会通过 requestId / selectedDateKey 校验丢弃，避免污染当前看板。
- 如果 `ai_day_journeys` 表不存在，页面会提示需要执行 `supabase/migrations/004_ai_day_journeys.sql`。

本轮重新生成：

- `docs/ai-day-journey/test-results/2026-06-03-generated-day-journey-v2.md`
- `docs/ai-day-journey/test-results/2026-06-04-generated-day-journey-v3.md`
- `docs/ai-day-journey/test-results/2026-06-04-generated-day-journey-v4.md`

上述结果均返回 `warnings: []`。

### 3.11 V1.5 段落组织链路

第 7.1 步后，「一天之旅」生成质量进入段落组织修复。

当前链路：

```text
ReviewBoard 触发生成
→ App.tsx 构造当天 events / tags payload
→ payload 携带 promptVersion: day-journey-system-prompt-v1.5
→ /api/ai/day-journey 读取 V1.5 系统提示词
```

### 3.12 V1.6 事实边界与一致性链路

第 7.2 步后，「一天之旅」进入同日期多次生成稳定性修复。

V1.6 的核心变化是重新划清事实来源：

```text
当前 payload / events / tags
→ 生成 current_day_fact_whitelist
→ 构造不含 Golden 具体事实的 style guide
→ /api/ai/day-journey 读取 V1.6 系统提示词
→ 模型只能使用当前日期 payload 中的事实
→ 接口检查是否出现 Golden 外来事实污染
→ 返回 markdown / warnings
→ ReviewBoard 展示
```

V1.6 链路约束：

- 当前日期 payload 是唯一事实来源。
- Golden md 只作为格式、风格、段落组织和 Eval Rubric 参考。
- 不再把 6 月 1 日到 6 月 4 日完整 Golden md 原文放进生成上下文。
- 如果输出出现当前 payload 之外的人名、地点、工具、项目、书名或事件，应进入 warnings 或 Badcase 记录。
- `temperature` 从 `0.2` 调整为 `0.1`，优先提升记录型生成的一致性。
- 保留 V1.5 的自然段合并规则、V1.4 的缩进规则和 V1.3 的事实保留规则。
