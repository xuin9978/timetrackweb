# 文科生够用工程概念词典

## 0. 文件定位

本文件用于记录我在实现「一天之旅」AI 功能过程中遇到的工程概念。

解释要求：
- 面向文科 / 商科背景用户。
- 不写过度技术化定义。
- 只解释实现这个 AI 功能所需的够用含义。
- 每个概念都要说明“它在本项目里出现在哪里”。

后续当我问到新的工程概念，或 CodeX 在实现中用到新的工程概念时，请主动补充到本文件。

## 1. 当前已知概念

| 术语 | 文科生够用解释 | 本项目里的位置 |
|---|---|---|
| Vite | 前端项目的开发工具，负责启动本地网站、处理开发环境和打包项目。 | `vite.config.ts` |
| config | 配置，意思是告诉工具这个项目应该怎么运行。 | `vite.config.ts` |
| ts | TypeScript 文件后缀，类似 `.js`，但有更严格的类型规则。 | `vite.config.ts`、`.tsx` 文件 |
| API 接口 | 前端页面和后台功能之间的通道。前端把数据发给接口，接口处理后返回结果。 | 当前 `/api/ai/daily-review`，未来 `/api/ai/day-journey` |
| AI 接口 | 专门用于调用大模型的 API 接口。它接收日程数据，调用模型，再把 AI 结果返回给前端。 | 当前 AI 日复盘接口，未来一天之旅接口 |
| 前端组件 | 页面上看得见、可交互的模块。 | `components/ReviewBoard.tsx` |
| 后端接口 | 接收前端请求、处理数据、调用服务、返回结果的后台通道。 | 当前临时写在 `vite.config.ts` |
| 系统提示词 | 后台固定给模型的规则，用来约束 AI 应该怎么生成内容。 | `docs/ai-day-journey/一天之旅系统提示词.md` |
| few-shot 样例 | 给 AI 看的少量参考范本，用来告诉 AI 什么样的输出才合格。 | `docs/ai-day-journey/examples/` |
| Markdown | 一种适合保存标题、段落、加粗、列表的文档格式。 | 一天之旅输出格式 |
| JSON | 一种结构化数据格式，适合前后端传递数据。 | AI 日复盘当前返回格式、未来可能用于一天之旅 |
| fetch | 前端向接口发送请求的方式。 | 当前 AI 日复盘请求逻辑 |
| 环境变量 | 存放 API Key 等敏感配置的地方，不应该直接写死在前端代码里。 | `GLM_API_KEY`、`DEEPSEEK_API_KEY` |
| 系统提示词稳定版 | 已经从草稿打磨到可以进入后台调用链路的提示词版本。它不只是写给人看的说明，而是后续可以直接约束模型输出的规则文本。 | `docs/ai-day-journey/一天之旅系统提示词.md` |
| Markdown 输出 | 让模型直接生成 `.md` 风格正文，适合复制、保存和下载。当前「一天之旅」V0 优先采用这种方式。 | `docs/ai-day-journey/一天之旅系统提示词.md` |
| JSON 输出 | 让模型先返回结构化字段，前端或后端再拼成页面内容。它更稳定，但实现更复杂。 | `docs/ai-day-journey/一天之旅系统提示词.md` 的后续可选方案 |
| JSON + Markdown | 一种更稳但更复杂的方案：模型返回结构化 JSON，同时包含或生成 Markdown，方便前端分块展示和检查。 | `docs/ai-day-journey/一天之旅系统提示词.md` |
| prompt_version | 提示词版本号。后续如果 AI 生成错了，可以知道当时用的是哪一版规则，方便复盘和修正。 | 当前建议 `day-journey-system-prompt-v1` |
| 接口契约 | 在真正写代码前，先约定接口地址、前端要传什么、后端要返回什么、失败时怎么返回。它像一份“前后端合作说明书”。 | `docs/ai-day-journey/04_接口_一天之旅接口契约设计.md` |
| payload | 前端发给接口的数据包。对「一天之旅」来说，payload 里会放日期、当天事件、标签和提示词版本。 | `docs/ai-day-journey/04_接口_一天之旅接口契约设计.md` |
| response | 接口返回给前端的数据包。对「一天之旅」来说，response 里会放生成是否成功、模型来源、Markdown 内容和错误信息。 | `docs/ai-day-journey/04_接口_一天之旅接口契约设计.md` |
| fallback | 兜底方案。比如 GLM 调用失败时，再尝试 DeepSeek；如果都失败，就返回明确错误，而不是让页面卡住。 | 当前 AI 日复盘接口已有 provider fallback 思路，未来一天之旅接口也会用到 |
| token | 模型处理文本时的计量单位，可以粗略理解为“模型读写文字的成本单位”。输入越长、输出越长，token 通常越多。 | `docs/ai-day-journey/04_接口_一天之旅接口契约设计.md` 的 `usage` 字段 |
| provider | 模型服务商，例如 GLM、DeepSeek。接口需要知道实际是哪家服务生成了结果。 | 当前 AI 日复盘接口和未来 `/api/ai/day-journey` |
| model | 具体使用的大模型名称，例如 `glm-4-flash` 或 `deepseek-chat`。provider 是服务商，model 是具体型号。 | 当前 AI 日复盘接口和未来 `/api/ai/day-journey` |
| API Route | 一个具体的接口地址。前端请求这个地址，后台就执行对应逻辑。 | `POST /api/ai/day-journey` |
| middleware | 开发服务器中间层。可以理解为网站请求到达页面前，先被这里拦一下并处理。 | `vite.config.ts` 里的 `/api/ai/daily-review` 和 `/api/ai/day-journey` |
| POST | 一种请求方式，通常用于“提交数据并让后台处理”。 | 前端后续调用 `/api/ai/day-journey` 时使用 POST |
| request body | POST 请求里真正装数据的部分，例如日期、events、tags。 | `/api/ai/day-journey` 接收的 JSON |
| status code | 接口返回的状态数字，用来表示请求是成功、参数错误还是服务失败。 | `/api/ai/day-journey` 会返回 200、400、405、500、502 等状态 |
| provider fallback | 模型服务兜底链路。先试 GLM，如果失败，再试 DeepSeek。 | `vite.config.ts` 的 `/api/ai/day-journey` |
| warnings | 生成成功但存在格式小问题时给前端的提醒，不等同于失败。 | 一天之旅 Markdown 缺少某些必要结构时，response 的 `warnings` 字段 |
| validation | 基础校验。接口先检查日期、记录、模型输出结构，避免明显错误直接进入页面。 | `/api/ai/day-journey` 的 date、events 和 Markdown 结构检查 |
| test payload | 专门用来测试接口的一包样例数据。它不是用户页面上的真实操作，而是为了验证接口能否处理某种输入。 | `docs/ai-day-journey/test-payloads/2026-06-04-day-journey-payload.json` |
| mock data | 模拟数据。不是从数据库实时读取，而是人为整理出来用于测试的固定样例。 | 2026 年 6 月 4 日接口测试 payload |
| curl | 一个从命令行请求接口的小工具，可以不用打开页面，直接测试接口是否能返回结果。 | 本轮用于测试 `/api/ai/day-journey` |
| local dev server | 本地开发服务。它让项目在本机跑起来，方便测试页面和本地 API。 | 本轮用本地服务测试 `/api/ai/day-journey` |
| 真实样例测试 | 用接近真实业务的数据测试接口，而不是只测空数据或随便写几条数据。 | 2026 年 6 月 4 日一天之旅接口测试 |
| Badcase | 不符合预期的输出案例。它不一定代表接口坏了，但代表生成质量还没达到标准。 | `docs/ai-day-journey/05_Eval_一天之旅生成验收与Badcase记录.md` |
| max_tokens | 给模型设置的最大输出长度。可以粗略理解为“最多允许 AI 写多少内容”。如果太小，长文可能写到一半被截断。 | `/api/ai/day-journey` 的模型调用参数 |
| 输出截断 | AI 结果写到一半突然结束，通常是输出长度不够或模型中途停止。 | 第 4.6 步第一次 v2 生成曾出现末尾截断，随后通过提高输出上限修复 |
| 可读事件清单 | 把 JSON 事件整理成更接近人能读懂的清单，让模型更容易理解时间、时段和原始事实。 | `/api/ai/day-journey` 的 `events_readable_for_generation` |
| 质量 warnings | 不是接口失败，而是提醒“生成结果可能有质量问题”。例如时间格式不对、一天回顾太短、结尾疑似截断。 | `/api/ai/day-journey` 的 `warnings` |
| loading state | 页面正在等待结果时的状态。用户点击「生成一天之旅」后，按钮变成“生成中...”，避免重复点击。 | `components/ReviewBoard.tsx` |
| error state | 页面遇到失败时的状态。比如当天没有记录、AI 失败或 API Key 没配好时，页面需要显示明确提示。 | `components/ReviewBoard.tsx` |
| props | 父组件传给子组件的数据。这里是 `App.tsx` 把当前选中日期、events、tags 传给 `ReviewBoard`。 | `App.tsx`、`components/ReviewBoard.tsx` |
| state | 组件自己记住的当前状态。这里包括是否生成中、是否成功、错误信息、Markdown 结果和 warnings。 | `components/ReviewBoard.tsx` |
| markdown rendering | 把 Markdown 内容展示成用户能直接阅读的样式，比如标题、段落、加粗，而不是直接显示 `##`、`###`、`**`。 | `components/ReviewBoard.tsx` |
| frontend integration | 前端接入。意思是把已经完成的后台接口接到真实页面按钮上，让用户能点击并看到结果。 | `components/ReviewBoard.tsx` 调用 `/api/ai/day-journey` |
| Golden Set | 一组高质量标准答案。它不是普通示例，而是后续判断 AI 生成结果是否合格的参考答案集合。 | `docs/ai-day-journey/examples/`、`05_Eval_一天之旅GoldenSet对比记录.md` |
| Golden Reference | 某一次测试要对齐的具体标准答案。例如 2026 年 6 月 3 日生成结果要对齐 6 月 3 日 Golden md。 | `/api/ai/day-journey` 在 V1.2 中按日期读取对应 Golden 文件 |
| Eval | 评测。意思是拿 AI 输出和 Golden Reference 比较，检查事实、格式、风格是否达标。 | `05_Eval_一天之旅生成验收与Badcase记录.md` |
| displayStartTime | 专门给 AI 输出正文使用的开始时间。它解决模型自己改时间的问题。 | `vite.config.ts`、`App.tsx`、`test-payloads/2026-06-03-day-journey-payload.json` |
| 时间点失真 | AI 把原始时间改错，例如把 `5:30` 写成 `6:00`。这是事实错误，不是风格问题。 | 2026 年 6 月 3 日 Badcase |
| 输入构造 | 把前端事件整理成模型更容易理解的数据。整理得越清楚，AI 越不容易漏事实或改错时间。 | `/api/ai/day-journey` 的 `events_json` 和 `events_readable_for_generation` |
| few-shot 参考 | 给模型看少量高质量例子，让它模仿格式、粒度和表达方式。 | V1.2 中按日期读取 Golden Reference |
| rawTitle | 原始标题。它保留用户原本记录的事件文字，避免模型只看到整理后的标题而漏掉细节。 | V1.3 的 `/api/ai/day-journey` 输入构造 |
| rawDescription | 原始描述。它保留事件的补充说明，供模型判断不能遗漏哪些事实。 | V1.3 的 `/api/ai/day-journey` 输入构造 |
| suggestedSegment | 建议分段。后台根据时间、标签和饭点先判断这条记录更像早上、下午还是晚上，供模型参考。 | V1.3 的 `/api/ai/day-journey` 输入构造 |
| mustKeepFacts | 必须保留事实。后台从原始标题和描述里拆出关键事实，提醒模型不能遗漏、不能改写。 | V1.3 的 `/api/ai/day-journey` 输入构造 |
| 事实改写 | AI 把原始事实写成另一个意思。例如“帮老爸弄照片”被写成“老爸生日”。这是质量问题。 | 6 月 2 日 Golden Set Badcase |
| 结构硬约束 | 某些标题或段落必须出现，不能缺失。比如「一天之旅」必须有 `### 一天回顾`。 | 系统提示词 V1.3 和接口 warnings |
| clipboard | 剪贴板。用户点击复制后，页面把当前 Markdown 放进系统剪贴板，用户就可以粘贴到别处。 | `components/ReviewBoard.tsx` 的「复制 Markdown」 |
| download blob | 浏览器里临时生成一个可下载文件。这里用当前 Markdown 临时生成 `.md` 文件，不需要先保存到服务器。 | `components/ReviewBoard.tsx` 的「下载 .md」 |
| disabled state | 按钮不可点击状态。比如 AI 正在生成时，复制、下载、重新生成按钮会暂时不可点，避免重复操作。 | `components/ReviewBoard.tsx` |
| retry | 重试。生成失败或用户想刷新结果时，再次调用同一个生成接口。 | `components/ReviewBoard.tsx` 的「重新生成」 |
| client-side download | 前端本地下载。文件直接在浏览器里生成并下载，不经过数据库保存。 | `components/ReviewBoard.tsx` |
| raw Markdown | 原始 Markdown 文本，里面保留 `##`、`###`、`**` 等符号。它适合复制、下载和保存为 `.md` 文件。 | `/api/ai/day-journey` 返回的 `markdown` 字段，`ReviewBoard` 的复制和下载逻辑 |
| rendered view | 渲染后的阅读视图。用户在页面里看到的是标题、段落和加粗效果，而不是 Markdown 符号本身。 | `components/ReviewBoard.tsx` 的展示区 |
| copy source | 复制来源。这里指点击「复制 Markdown」时，复制的是原始 Markdown，而不是页面上渲染后的样式。 | `components/ReviewBoard.tsx` 的复制逻辑 |
| display layer | 展示层。它只负责把数据展示得更好读，不改变接口返回的数据，也不改变复制和下载用的原文。 | `components/ReviewBoard.tsx` 的 Markdown 渲染逻辑 |
| 中文全角空格 | 中文排版里占一个汉字宽度的空格。这里要求每个正文段落开头有两个 `　　`，让「一天之旅」更像正式日记 / 时间账本。 | `docs/ai-day-journey/一天之旅系统提示词.md` V1.4 |
| 段落首行缩进 | 正文段落第一行开头空出固定距离。这里不是前端随便加样式，而是要求 AI 输出的原始 Markdown 本身带有 `　　`。 | 系统提示词 V1.4、`/api/ai/day-journey` warnings |
| whitespace-pre-wrap | 一种前端显示方式，可以保留文本里的空格和换行。这里用来确保页面展示时不吞掉 `　　` 缩进。 | `components/ReviewBoard.tsx` |
| 格式硬约束 | 必须遵守的格式规则，不是建议。比如正文段落必须以两个中文全角空格开头。 | 系统提示词 V1.4 |
| persistence | 持久化保存。意思是内容不只停留在页面上，而是保存到数据库里，刷新或切换日期后还能读回来。 | `ai_day_journeys` 表、`utils/dayJourneyService.ts` |
| upsert | 有就更新，没有就新增。这里同一用户同一天已经保存过「一天之旅」时，再点保存会更新原记录，而不是新增第二条。 | `utils/dayJourneyService.ts` 的 `saveDayJourney` |
| unique constraint | 唯一约束。它规定某些字段组合不能重复。这里是 `user_id + date` 不能重复，保证每天只有一份最新版。 | `supabase/migrations/004_ai_day_journeys.sql` |
| RLS | 行级安全规则。可以理解为数据库里的权限门卫，确保用户只能看到和修改自己的记录。 | `ai_day_journeys`、`events`、`tags` 的 Supabase policy |
| migration | 数据库变更文件。它记录要新增什么表、字段和权限。创建文件不等于线上数据库已生效，还需要执行。 | `supabase/migrations/004_ai_day_journeys.sql` |
| saved state | 已保存状态。表示当前看板内容已经写入当前日期的数据库记录。 | `App.tsx`、`ReviewBoard.tsx` 的保存提示 |
| dirty state | 未保存状态。表示页面上有新生成内容，但还没有点「保存」写入数据库。 | `App.tsx` 的 `dayJourneyIsDirty` |
| load saved content | 读取已保存内容。切换日期时，系统会查这个日期是否保存过「一天之旅」，有就显示，没有就清空。 | `App.tsx` 调用 `getDayJourney` |
| inputSnapshot | 生成输入快照。保存这次生成时用过的 date、events、tags、模型信息和 warnings，方便以后做 Eval、Badcase 和追踪。 | `ai_day_journeys.input_snapshot` |
