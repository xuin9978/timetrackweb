# ai-day-journey 目录 README

## 1. 目录定位

`docs/ai-day-journey/` 是「一天之旅」AI 功能的 **最终落地文档区**。

这个目录主要服务于以下几类任务：

- 回顾这个 AI 功能是怎么从 0 到 1 落地的
- 理解当前真实实现状态
- 查看 Prompt、Golden、Eval、Badcase 和接口链路
- 为作品集、简历、面试复盘提供证据材料
- 为后续继续迭代该功能提供上下文

这个目录不是单纯的资料堆放区，而是：

> 「一天之旅」功能的产品定义、实现过程、质量评估和复盘证据中心

---

## 2. 建议和哪个目录搭配看

请和 `docs/ai-product/` 搭配阅读。

二者区别如下：

- `docs/ai-product/`
  - 看“为什么要做这个 AI 功能”
  - 偏早期探索、产品规划、方向收敛

- `docs/ai-day-journey/`
  - 看“这个 AI 功能最后怎么做出来了”
  - 偏最终落地、Prompt、接口、Eval、Badcase、实现状态

建议顺序：

1. 先看 `docs/ai-product/`
2. 再看 `docs/ai-day-journey/`

## 4. 各类文件分别是干什么的

### 总览类

- 帮助快速建立上下文
- 适合第一次接触这个功能的人

### PRD 类

- 解释这个功能从产品角度是怎么被定义的

### Prompt 类

- 解释模型为什么会按现在这个格式生成
- 记录版本变化和修复逻辑

### Golden / 样例类

- 最高优先级输出标准
- Eval 和 Badcase 的对照依据

### 接口类

- 解释前端、接口、模型、保存之间的真实链路

### Eval 类

- 解释为什么“能生成”不等于“做完了”
- 用于定位质量问题、记录修复过程

### 测试类

- 保存真实测试输入与输出
- 便于回放和复盘

### 面试 / 作品集类

- 把这个项目转译成更适合表达和展示的材料

## 6. 阅读时需要注意的边界

1. 不要把早期探索文档等同于最终已实现能力。
2. 不要把 Golden 样例中的具体事实误当成其他日期的事实来源。
3. 当前真实规则应优先以 `一天之旅系统提示词.md` 为准。
4. Eval / Badcase / Consistency Eval 比单次成功结果更能代表功能成熟度。
5. 如果后续继续迭代，应同时关注：
   - Prompt
   - 输入构造
   - 输出质量
   - 前端展示
   - 保存链路
   - Eval 记录
