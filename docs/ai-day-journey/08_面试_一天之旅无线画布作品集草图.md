# 一天之旅无线画布作品集草图

## 0. 文件定位

本文件用于把「一天之旅」AI 功能整理成适合无线画布展示的作品集结构。

目标不是写成传统 PRD，而是帮助我把：

- 页面截图
- 产品说明
- Prompt / Eval / Badcase 证据
- 调用链路
- 保存链路
- 项目复盘

组织成一份适合面试展示、作品集投递和口头讲解的画布型材料。

本文件默认面向：

- AI 产品经理校招 / 实习面试
- 项目复盘展示
- 作品集链接展示

---

## 1. 总体展示原则

这份画布不要做成“截图堆叠”。

更好的方式是：

- 每个区块都回答一个明确问题。
- 左边放结论，右边放证据。
- 截图旁边写产品判断，而不是功能介绍。
- 让面试官顺着画布一路看下去时，能自然理解：
  - 你为什么做这个功能
  - 你怎么定义它
  - 你怎么把它做出来
  - 你怎么验证它
  - 你怎么迭代它

一句话主线建议：

> 我把一个已有时间管理工具，从“记录时间”升级成了“生成、评估、保存 AI 日复盘资产”的产品。

---

## 2. 画布总结构

推荐把画布拆成 8 个区块：

1. 项目总览
2. 问题与机会
3. 功能收敛
4. 核心方案
5. Workflow 与调用链路
6. Prompt / Eval / Badcase
7. 保存与用户资产
8. 项目复盘与能力总结

如果画布足够大，可以横向排布。

推荐阅读顺序：

```text
项目是什么
→ 为什么值得做 AI
→ 为什么最后选一天之旅
→ 方案长什么样
→ 怎么跑起来
→ 怎么做质量控制
→ 怎么沉淀成用户资产
→ 我作为 AI PM 做了什么判断
```

## 3. 区块 1：项目总览

### 目标

让对方 10 秒内知道：

- 这是什么项目
- 原产品是什么
- AI 功能加进去后解决了什么

### 建议放的内容

- 项目名称
- 一句话定位
- 一张总览截图
- 一张 AI 看板截图

### 建议标题

`时间管理项目：从记录时间到生成 AI 日复盘资产`

### 建议配文

可以写成 3 行以内：

> 这是一个原本已经具备日历、标签、计时器、历史记录和 Supabase 同步的时间管理工具。  
> 我在这个基础上新增了「一天之旅」AI 功能，让系统基于当天时间记录自动生成 Markdown 日复盘，并支持按日期保存和回查。  
> 这个项目的重点不是“接一个模型”，而是完成从功能定义、Prompt、Eval 到用户资产沉淀的闭环。

## 4. 区块 2：问题与机会

### 目标

说明为什么原产品虽然能记录时间，但还不够。

### 建议放的截图

- 原始日历页面截图
- 原始事件列表截图
- 历史记录页面截图

### 旁边要写的核心判断

不要写“这里可以查看日历”。

更好的写法是：

> 原产品已经解决了“记录时间”和“查看时间”的问题，但用户仍然需要自己把原始事件整理成可回看的复盘内容。  
> 也就是说，系统能告诉你“今天做了什么”，但还不能把这些记录转成一份可阅读、可回查、可沉淀的时间资产。

### 面试时要讲的点

- 这是一个已有数据基础的产品升级，不是凭空做 AI 套壳。
- AI 的价值不在生成漂亮文案，而在降低用户从“记录”走到“理解”的成本。

## 5. 区块 3：功能收敛

### 目标

体现你不是直接做「一天之旅」，而是从多个 AI 方向中做了选择。

### 建议放的内容

- 一个小表格或小流程图
- 候选方向：
  - AI 自动日复盘
  - AI 周复盘
  - 标签分析
  - 异常提醒
  - 明日建议
  - 时间记录质量检查

### 要传达的核心判断

> 我最初考虑的是更泛的 AI 自动复盘，但在实际落地时发现，开放式总结不够可评估，也不够有辨识度。  
> 所以我把功能收敛成了「一天之旅」：固定结构、固定风格、可做 Golden Set、可做 Badcase、也更适合作品集展示。

## 6. 区块 4：核心方案

### 目标

展示「一天之旅」最后长什么样。

### 建议放的截图

- 一天之旅看板截图
- 生成后的正文截图
- 如果空间允许，可以放 2026-06-04 最终范本局部截图

### 应该强调的结构

- 日期
- 关键词
- ☀️早上 / 🌇下午 / 🌃晚上
- `◆` 阶段归纳
- `### 一天回顾`

### 建议文案

> 我最终没有让 AI 输出一个泛泛的“总结 + 建议”，而是规定它生成一种接近柳比歇夫式时间账本的固定格式。  
> 这样做的好处是：结构稳定、事实可核对、风格可迭代、质量可评估。

## 7. 区块 5：Workflow 与调用链路

### 目标

证明这不是“文档型概念项目”，而是真的跑起来了。

### 建议放的内容

- 一张流程图
- 一张接口链路说明截图
- 一张前端触发生成的页面截图

### 推荐流程图内容

```text
用户选择日期
→ 前端筛选当天 events / tags
→ POST /api/ai/day-journey
→ 后端读取系统提示词
→ 调用 GLM / DeepSeek
→ 返回 Markdown
→ 前端渲染结果
→ 用户点击保存
→ upsert 到 Supabase
```

### 建议文案

> 这个功能的技术形态是 LLM + Workflow。  
> 前端负责收集和筛选当天数据，后端负责规则化输入、调用模型和做基础校验，最终前端展示并支持保存。  
> 我刻意没有把它做成 Agent，因为第一版更重要的是稳定、可控、可验证。

## 8. 区块 6：Prompt / Eval / Badcase

### 目标

这块是整个作品集最有辨识度的部分。

### 建议放的内容

- Prompt 版本迭代图
- Golden Set 样例截图
- Badcase 前后对比截图
- Consistency Eval 摘要

### 一定要讲的几类 Badcase

- 时间点失真
- 关键词过泛 / 过细
- 一条 event 一个段落
- 一天回顾过短
- Golden 事实污染
- 同一日期多次生成漂移

### 建议文案

> 我没有把“能生成”当成完成，而是把质量控制当成产品的一部分。  
> 为了让结果可评估，我先整理了 4 个 Golden Case，再围绕事实一致性、分段、关键词、段落组织和回顾结构持续记录 Badcase。  
> 后面还做了同日期多次生成的一致性测试，专门修复输出漂移和事实污染。

## 9. 区块 7：保存与用户资产

### 目标

突出你已经从“一次性生成”走到了“用户资产沉淀”。

### 建议放的截图

- 生成后未保存状态截图
- 点击保存后的状态截图
- 切换日期后自动加载 / 自动清空的截图
- 如果方便，可以放一张 Supabase 表结构简图

### 建议文案

> 我没有把“一天之旅”停留在一次性生成结果，而是继续往前走了一步：让它支持按用户、按日期保存。  
> 这样它不再只是一个 AI Demo，而是开始具备“用户资产”的形态。

### 这里最值得强调的设计点

- `user_id + date` 唯一约束
- upsert 更新同一天最新版
- selectedDateKey 防止时区错位
- requestId 防止异步污染
- 未保存日期自动清空
- 已保存日期自动加载

## 10. 区块 8：项目复盘与能力总结

### 目标

这一块是最像“面试结尾总结”的区域。

### 建议放的内容

- 我做对了什么
- 我踩过什么坑
- 如果做 V1 会继续做什么
- 这个项目体现了哪些 AI PM 能力

### 推荐写法

#### 我做对了什么

- 没有一开始就做复杂 Agent，而是先做 LLM + Workflow。
- 没有直接追求“生成得像人话”，而是先建立 Golden Set 和 Eval 标准。
- 没有把结果停留在页面，而是继续推进保存和回查链路。

#### 我踩过的坑

- 过早把 Golden 具体事实放进上下文，导致事实污染。
- 一开始段落组织太碎，像日历流水账。
- 同一日期多次生成时输出不够稳定。

#### 如果继续做 V1

- 增强一天回顾信息密度。
- 增加保存后的历史版本比较。
- 做更自动化的 Eval / 回归检查。

#### 这个项目体现的能力

- 选题与收敛能力
- PRD 和功能定义能力
- Prompt 与输入构造能力
- Eval / Badcase / Consistency Eval 意识
- 与工程协作和落地能力
- 用户资产化与保存闭环意识

## 11. 每个区块的统一模板

为了让整张画布更整齐，建议每一块都尽量使用同一个结构：

1. 标题
2. 一句话结论
3. 截图 / 图示
4. 3 到 5 行说明
5. 一条“我的判断”

例如：

```text
标题：为什么不是普通 AI 总结，而是一天之旅
一句话结论：我把开放式复盘收敛成固定结构，是为了提升可评估性和可迭代性。
截图：最终生成结果截图 + Golden 范本局部截图
说明：展示结构、分段、关键词、◆ 归纳、一天回顾
我的判断：固定输出不是限制模型，而是为了把 AI 功能变成产品能力
```

## 13. 最终建议

如果你的目标是 AI 产品经理面试，这份画布最重要的不是“好看”，而是“让别人快速看出你不是只会做页面”。

所以整份作品集要尽量让面试官看到这 4 件事：

1. 你有产品问题判断，而不是只会做功能。
2. 你理解 LLM 能力边界，而不是只会写 Prompt。
3. 你有 Eval 和 Badcase 思维，而不是只会展示成功结果。
4. 你有把 AI 功能沉淀成真实产品能力的意识，而不是停留在 Demo。

这才是这份作品集最有价值的部分。
