# 一天之旅接口契约设计

## 0. 文件定位

本文件用于说明 `/api/ai/day-journey` 接口应该如何设计。

本文最初创建于接口实现前，用于约定前后端输入、输出、错误兜底和保存策略。当前 `/api/ai/day-journey` 已在本地 Vite middleware 中实现，并已接入「一天之旅」前端看板；本文保留为接口契约源文档，用于回顾设计依据和后续 V1 迭代。

## 1. 接口定位

`/api/ai/day-journey` 是前端「一天之旅」看板和后端 LLM 调用之间的接口。

前端负责把当前日期、当天日程记录、标签信息和生成选项传给接口。接口负责校验输入、读取或引用「一天之旅系统提示词」、组织模型输入、调用大模型生成「一天之旅」Markdown，并把结果返回给前端展示。

## 2. 推荐接口路径

```text
POST /api/ai/day-journey
```

## 3. 请求 Payload 设计

### 3.1 推荐请求结构

```json
{
  "date": "2026-06-04",
  "timezone": "Asia/Shanghai",
  "events": [
    {
      "id": "event_id",
      "title": "泡咖啡、复盘昨日，并优化改进时间管理项目",
      "start": "2026-06-04T07:15:00+08:00",
      "end": "2026-06-04T08:00:00+08:00",
      "tagId": "tag_id",
      "tagName": "早上",
      "category": "☀️"
    }
  ],
  "tags": [
    {
      "id": "tag_id",
      "name": "早上",
      "color": "bg-blue-500",
      "icon": "☀️"
    }
  ],
  "options": {
    "promptVersion": "day-journey-system-prompt-v1",
    "outputFormat": "markdown"
  }
}
```

### 3.2 与当前项目类型的字段对应

当前项目里的事件类型是 `CalendarEvent`，定义在 `types.ts`：

```ts
export interface CalendarEvent {
  id: string;
  title: string;
  startTime: string;
  endTime: string;
  category: string;
  date: Date;
  description?: string;
}
```

接口 payload 字段建议这样映射：

| Payload 字段 | 当前项目来源 | 说明 |
| --- | --- | --- |
| `events[].id` | `CalendarEvent.id` | 事件 ID。 |
| `events[].title` | `CalendarEvent.title` | 原始事件标题，是生成「一天之旅」的核心事实来源。 |
| `events[].start` | `CalendarEvent.date` + `CalendarEvent.startTime` | 建议由前端或接口层转换成 ISO 字符串。当前事件本身没有 `start` 字段。 |
| `events[].end` | `CalendarEvent.date` + `CalendarEvent.endTime` | 建议由前端或接口层转换成 ISO 字符串。当前事件本身没有 `end` 字段。 |
| `events[].tagId` | `CalendarEvent.category` | 当前 `category` 实际引用 `Tag.id`。 |
| `events[].tagName` | `tags.find(tag.id === event.category)?.label` | 当前事件表不直接保存标签名，需要从 `tags` 映射。 |
| `events[].category` | `Tag.icon` 或 `Tag.label` | 当前没有独立 category 文案字段，V0 可用 tag icon / label 帮助模型理解。 |
| `tags[].id` | `Tag.id` | 标签 ID。 |
| `tags[].name` | `Tag.label` | 当前类型里叫 `label`，payload 中可转成更通用的 `name`。 |
| `tags[].color` | `Tag.color` | 当前是 Tailwind class，例如 `bg-blue-500`，不是十六进制色值。 |
| `tags[].icon` | `Tag.icon` | 可选 emoji，用于辅助早中晚和类别理解。 |

注意：不要假设当前数据库里已有 `description`、`tagName`、`start`、`end` 等字段。它们应由前端或接口层从当前 `CalendarEvent` / `Tag` 结构整理出来。

### 3.3 options 字段

| 字段 | 建议值 | 说明 |
| --- | --- | --- |
| `promptVersion` | `day-journey-system-prompt-v1` | 标记本次生成使用的系统提示词版本。 |
| `outputFormat` | `markdown` | V0 优先直接返回 Markdown。 |

## 4. Response 设计

### 4.1 成功返回

V0 阶段优先返回 Markdown：

```json
{
  "success": true,
  "provider": "glm",
  "model": "glm-4-flash",
  "promptVersion": "day-journey-system-prompt-v1",
  "date": "2026-06-04",
  "markdown": "## 2026 年 6 月 4 日\n\n　　**关键词：...**",
  "warnings": [],
  "usage": {
    "inputTokens": null,
    "outputTokens": null
  }
}
```

字段说明：

| 字段 | 说明 |
| --- | --- |
| `success` | 是否生成成功。 |
| `provider` | 实际调用的模型服务商，例如 `glm`、`deepseek`。 |
| `model` | 实际模型名。 |
| `promptVersion` | 本次使用的系统提示词版本。 |
| `date` | 本次生成对应的日期。 |
| `markdown` | 生成出的「一天之旅」Markdown 正文。 |
| `warnings` | 非阻断问题，例如记录较少、某个时段无记录、格式轻微修正。 |
| `usage` | token 用量。部分 provider 可能没有返回，V0 可为 `null`。 |

### 4.2 失败返回

```json
{
  "success": false,
  "error": {
    "code": "AI_GENERATION_FAILED",
    "message": "一天之旅生成失败，请稍后重试"
  }
}
```

建议错误码：

| code | 场景 |
| --- | --- |
| `NO_EVENTS` | 当天没有日程记录。 |
| `INVALID_PAYLOAD` | 缺少 date、timezone、events 等必要字段，或字段格式错误。 |
| `MISSING_API_KEY` | 没有配置可用的 LLM API Key。 |
| `PROVIDER_FAILED` | 单个 provider 调用失败。 |
| `AI_GENERATION_FAILED` | 所有 provider 都失败。 |
| `EMPTY_MODEL_OUTPUT` | 模型返回空内容。 |
| `INVALID_MARKDOWN_FORMAT` | 模型输出不是合格 Markdown，或缺少关键结构。 |

## 5. 为什么 V0 先返回 Markdown

V0 建议直接返回 Markdown，原因是：

- 「一天之旅」本身是文档型输出。
- 用户需要复制、保存、下载为 `.md`。
- 当前 V0 先直接返回 Markdown，便于快速跑通功能。
- 直接 Markdown 能更快验证事实保留、格式稳定和最终范本风格是否一致。
- 后续如果前端展示需要更稳定，可以升级为 JSON + Markdown 双格式。

## 6. 接口内部 Workflow

```text
接收前端 payload
→ 校验 date / events
→ 如果当天没有记录，返回空状态
→ 读取或引用 一天之旅系统提示词.md
→ 将当天原始日程记录整理成模型输入
→ 调用 GLM / DeepSeek / OpenAI-compatible API
→ 得到 Markdown
→ 按验收规则做基础检查
→ 返回 Markdown 给前端
```

## 7. 错误兜底设计

| 场景 | 建议处理 |
| --- | --- |
| 当天没有日程记录 | 不调用 LLM，返回 `success: false`、`code: NO_EVENTS`，前端展示“当天暂无可生成的一天之旅记录”。 |
| LLM API Key 缺失 | 返回 `code: MISSING_API_KEY`，不要把 Key 或环境变量值暴露给前端。 |
| GLM 调用失败 | 记录失败原因，尝试 DeepSeek fallback。 |
| DeepSeek 调用失败 | 如果 GLM 也失败，返回 `code: AI_GENERATION_FAILED`。 |
| 模型返回空内容 | 返回 `code: EMPTY_MODEL_OUTPUT`，前端允许重试。 |
| 模型输出格式不符合 Markdown | 返回 `code: INVALID_MARKDOWN_FORMAT`，或在后端做轻量格式检查后要求前端展示失败。 |
| 生成结果遗漏关键结构，例如没有 `### 一天回顾` | 判为 `INVALID_MARKDOWN_FORMAT`，不直接当作成功结果展示。 |

V0 不建议在失败时编造本地「一天之旅」。可以展示明确失败态和重试入口，避免用户误以为 AI 已成功生成。

## 8. 与现有 AI 日复盘接口的关系

当前项目已有 `/api/ai/daily-review`，它写在 `vite.config.ts` 的 Vite dev server middleware 中。

可复用内容：

- OpenAI-compatible chat completions 调用方式。
- 从环境变量读取 API Key 的方式。
- GLM 优先、DeepSeek 兜底的 provider fallback 思路。
- 解析模型返回内容、处理 provider 失败的基本流程。

建议调整：

- 不建议长期把新的 AI 功能继续堆在 `vite.config.ts`。
- V0 可以临时参考当前实现方式，但长期建议拆出公共 `llmClient`。
- 公共调用逻辑可抽成类似 `llmClient` 的模块，负责 provider 配置、fallback、错误归一化和 token usage 提取。
- `/api/ai/day-journey` 应有独立 prompt、独立 payload、独立 response，不要复用 AI 日复盘的 JSON 输出结构。

长期方向：

- 如果项目继续部署到生产环境，应迁移到正式后端 API 或 serverless function。
- 当前 Vite middleware 更适合本地 Demo 和开发验证。

## 9. 是否保存到 Supabase

本轮只设计，不实现。

当前建议：

- V0 可以先只生成并展示，不保存。
- V1 可考虑新增 `ai_day_journeys` 表保存生成结果。

如果后续保存，建议字段包括：

| 字段 | 说明 |
| --- | --- |
| `id` | 生成结果 ID。 |
| `user_id` | 用户 ID。 |
| `date` | 对应日期。 |
| `source_event_ids` | 本次生成引用的事件 ID 列表。 |
| `input_snapshot` | 生成时使用的输入快照，便于复现。 |
| `generated_content` | 生成出的 Markdown 或结构化内容。 |
| `prompt_version` | 使用的系统提示词版本。 |
| `model_provider` | 模型服务商。 |
| `model_name` | 模型名。 |
| `created_at` | 创建时间。 |
| `updated_at` | 更新时间。 |

这属于 Supabase schema 改动，后续需要单独确认。

## 10. prompt_version 设计

建议本接口固定传入并返回：

```text
promptVersion: day-journey-system-prompt-v1
```

用途：

- 追踪不同系统提示词版本。
- 后续做 Eval 和 Badcase 对比。
- 作品集展示 Prompt 迭代过程。
- 方便判断某次生成结果是由哪个提示词版本产生的。
- 当生成结果出现事实遗漏、格式漂移或表达不稳定时，可以定位问题是否来自 Prompt 版本。

## 11. V0 基础验收规则

后端拿到模型输出后，建议至少检查：

- 是否包含 `## 2026 年 X 月 X 日`。
- 是否包含 `**关键词：`。
- 是否包含 `### ☀️早上`。
- 是否包含 `### 🌇下午`。
- 是否包含 `### 🌃晚上`。
- 是否包含 `### 一天回顾`。
- 是否包含 `◆`。
- 是否返回非空 Markdown。

更细的事实覆盖率检查可以放到后续 Eval 阶段。
