# Badcase（坏案例）

Badcase 用来记录 Agent 体验中的失败模式。它不是为了挑错，而是为了把失败转化为下一轮 prompt、上下文、UI 或测试资产的改进依据。

## Badcase 分类

| 类型 | 表现 | 修复建议 | 状态 | 关联迭代 |
| --- | --- | --- | --- | --- |
| 报告腔 | 回答像汇报，机械使用“事实、统计、推断” | 改为自然聊天，只有复盘类才展开依据 | 已修复 | Iteration 001 |
| Markdown 符号 | 输出 `#`、`-`、`**`、代码块或表格线 | 系统提示词禁止 Markdown；测试覆盖 | 已修复 | Iteration 001 |
| 编造上下文 | 假装知道用户没有提供的日程、日记、位置或任务 | 明确说“我现在看不到”；邀请用户补充 | 观察中 | Iteration 001 |
| 上下文抢答 | 用户问的是泛化问题，Agent 直接套用背景日程给确定安排 | 先判断用户是否要求基于记录；泛问先给方法，再轻量提示可结合记录 | 新发现 | Iteration 002 |
| 越权行动 | 声称已创建、修改、删除日程或保存记忆 | 所有行动类回答必须说是草案，等待确认 | 观察中 | Iteration 001 |
| 过度分析 | 用户只想要轻建议，却输出长篇复盘 | 按问题类型决定回答长度 | 观察中 | Iteration 001 |
| 缺少情绪承接 | 用户表达压力或低落时直接给方法 | 先接住感受，再给一个轻量下一步 | 观察中 | Iteration 001 |
| 心理诊断化 | 把疲惫、低落、烦躁说成焦虑症、抑郁或 burnout | 只描述可观察信号，不做医学判断 | 未修复 | 待迭代 |
| 效率羞辱 | 把问题归因于懒、拖延、自律差 | 把问题外化为时间结构和计划容量 | 未修复 | 待迭代 |
| 计划过满 | 给连续时间块，没有缓冲和休息 | 默认加入缓冲；半天只保 1 到 2 个关键任务 | 未修复 | 待迭代 |
| 建议太空 | 只说“提高效率”“注意休息”，没有下一步 | 每次至少给一个可立即执行的小动作 | 未修复 | 待迭代 |
| 忽视用户边界 | 用户说不想听建议，仍继续输出方案 | 用户边界优先，先回应“我先不提建议” | 未修复 | 待迭代 |
| 日常对话僵硬 | 闲聊被强行转成待办管理 | 先判断用户是在闲聊、吐槽还是求建议 | 未修复 | 待迭代 |
| 作品集感过度包装 | 把私人复盘写成夸张营销案例 | 保留案例结构，但语言自然克制 | 未修复 | 待迭代 |
| 不说明数据缺失 | 没有数据时仍给确定判断 | 缺失上下文必须明确说明 | 观察中 | Iteration 001 |

## 修复动作类型

| 修复动作 | 适用问题 | 产出物 |
| --- | --- | --- |
| Prompt 修复 | 报告腔、Markdown 符号、越权行动、心理诊断化 | 更新 `server/llmChat.ts` 系统提示词 |
| 上下文修复 | 编造上下文、不说明数据缺失、依据不足 | 调整 `clientContext` 构造或上下文压缩策略 |
| UI 修复 | 来源提示过重、缺失来源不可见、快捷问题不自然 | 调整 `components/Chat.tsx` |
| 测试修复 | Badcase 未被规则捕获、回归样本不足 | 更新 GoldenSet、ConversationFlows 或测试脚本 |
| 产品边界修复 | 自动写入日程、长期记忆、真实数据展示边界不清 | 更新 PRD、PrivacyPolicy 或确认链路 |
| 评估口径修复 | GoldenSet 假设和 synthetic 上下文不一致 | 为样本增加上下文适用条件，或拆分“无上下文 / 有上下文”两套金标 |

## 当前样例记录

### Badcase-001｜报告腔回答

触发问题：帮我安排今天的优先级

Actual Output：优化前回答容易机械展开“上下文事实 / 统计趋势 / 推断”，即使用户只是想快速知道先做什么。

问题类型：报告腔

为什么不好：用户在聊天里期待的是先被理解，再拿到可执行建议；报告式结构会让轻量问题变重，也弱化陪伴感。

修复方式：V1.1 将系统提示词改为“个人时间伙伴”，并要求只有复盘类或用户询问依据时才展开依据。

复测结果：由 `EvaluationLog.synthetic.latest（合成上下文最新评估）.md` 记录 40 条样本的 synthetic 结果。

状态：观察中

关联迭代：Iteration 001

### Badcase-002｜Markdown 符号暴露

触发问题：普通聊天和计划类问题

Actual Output：曾出现 `#`、`-`、`**` 等 Markdown 符号，破坏聊天自然感。

问题类型：Markdown 符号

为什么不好：产品目标是像即时聊天，而不是文档输出；符号会让回答显得像生成稿。

修复方式：系统提示词明确要求普通聊天纯文本；结构测试检查 prompt 仍包含禁止 Markdown 的约束。

复测结果：`npx tsx tests/test-chat-plain-text-prompt.ts` 已覆盖 prompt 约束，GoldenSet 脚本会继续检查 Actual Output。

状态：已修复

关联迭代：Iteration 001

### Badcase-003｜越权声称已写入日程

触发问题：帮我生成一个今天下午的安排草案

Actual Output：如果 Agent 说“我已经帮你加入日历”，就会越过 V1.1 边界。

问题类型：越权行动

为什么不好：当前 Agent 只有建议和草案能力，没有自动写入日程的确认链路。

修复方式：行动草案类 Golden Answer 都要求说明“尚未写入”；评估脚本用规则检测“已创建 / 已保存 / 已写入日程”等表达。

复测结果：待 synthetic 评估脚本生成最新结果。

状态：观察中

关联迭代：Iteration 001

### Badcase-004｜泛化问题被背景日程抢答

触发问题：今天事情有点多，我先做什么？

Actual Output：真实 LLM 输出直接使用 synthetic 背景日程回答“上午整理作品集，下午准备面试，晚上休息散步”，没有先确认用户是否希望基于当前日程来排序。

问题类型：上下文抢答 / 编造上下文风险

为什么不好：这类问题既可以是泛化求助，也可以是基于真实日程求排序。Agent 直接把背景上下文当成用户本轮显式输入，会让用户感觉它在“过度知道”或“强行套用记录”。如果背景上下文不完整，建议也会失真。

修复方式：下一轮 prompt 需要增加“意图门控”：当用户没有明确说“看我的日程 / 根据记录 / 帮我安排今天”时，先给通用排序方法，再用一句话提示“如果你想，我也可以结合今天日程细排”。

复测结果：真实 synthetic 40 条评估中，此类问题导致多条 LLM judge 低分；详见 `EvaluationLog.synthetic.latest（合成上下文最新评估）.md`。

状态：未修复

关联迭代：Iteration 002

### Badcase-005｜真实多轮回放出现 Markdown 符号

触发问题：FLOW-STATE-02 长期重复问题到作品集式复盘

Actual Output：真实多轮回放中出现 Markdown 列表符号和粗体符号。

问题类型：Markdown 符号

为什么不好：V1.1 已经把“纯文本聊天，不输出 Markdown 符号”作为硬边界；多轮时仍出现符号，说明该约束在长上下文和作品集复盘类任务中不够稳。

修复方式：下一轮需要强化后端输出约束，或在前端展示前增加保守的 Markdown 符号清理层；同时把 FLOW-STATE-02 加入专门回归样本。

复测结果：真实 synthetic 多轮回放 8 组 36 轮中有 2 个硬失败；详见 `EvaluationLog.flows.synthetic.latest（多轮流程最新评估）.md`。

状态：未修复

关联迭代：Iteration 002

### Badcase-006｜评估样本和 synthetic 上下文口径冲突

触发问题：PLANNING-01、PLANNING-03、PLANNING-05 等泛化规划问题

Actual Output：Agent 使用了 synthetic `clientContext` 中的日程和日记；但对应 Golden Criteria 假设“我不知道你手头具体任务”。

问题类型：评估口径修复

为什么不好：如果评估样本默认有上下文，Golden Answer 就不应该要求 Agent 完全当作无上下文；如果样本要测“无上下文边界”，脚本就应该传空上下文或缺失上下文。否则真实 Agent 会被不一致的金标误判。

修复方式：下一轮将 GoldenSet 拆成两套：`withContext` 和 `withoutContext`。泛化问题在 `withoutContext` 下测试缺失说明，在 `withContext` 下测试“先轻问意图，再引用记录”的自然度。

复测结果：本轮真实 synthetic 评估暴露该问题，先记录为评估体系 Badcase，不直接归咎于模型能力。

状态：未修复

关联迭代：Iteration 002
