# 个人时间 Agent 聊天记录持久化计划

## 1. 目标

本计划只解决“第二层：聊天记录持久化”，让个人时间 Agent 具备会话级多轮记忆能力。

完成后，用户刷新页面、重新进入聊天模块，仍然可以看到历史会话，并继续上一次对话。模型请求时可以带上当前会话最近消息，使 Agent 能理解本轮对话里已经讨论过的任务、偏好和上下文。

这一版不做长期记忆、用户画像、RAG、工具调用、自动写日程，也不自动提取偏好。

## 2. 当前问题

当前 V1 已经具备短期多轮能力：前端内存中保存 `messages`，每次请求会把最近对话消息传给 `/api/chat`。

但它还不是真正的多轮记忆，原因是：

- 刷新页面后聊天记录丢失。
- 用户无法查看历史会话。
- 不能继续上一次对话。
- 聊天内容没有保存到 Supabase 或本地持久层。
- 模型只能理解当前页面内存中的消息，无法理解历史会话。

## 3. 产品范围

### 3.0 已锁定产品决策

- 数据层：采用 Supabase 云端持久化 + localStorage fallback。
- 会话列表：做完整历史会话列表，不只做最近会话恢复。
- 会话栏位置：放在聊天模块内部，位于主导航栏右侧、聊天主内容区左侧。
- 会话栏宽度：约 260px。
- 会话标题：第一版用第一条用户消息截断生成，同时支持手动重命名。
- 会话列表信息：显示标题 + 更新时间，不显示最后一条消息摘要。
- 更新时间格式：使用相对时间，如“刚刚”“5分钟前”“今天 14:30”“昨天 14:30”“07-18”。
- 新对话：点击后先进入空白聊天页，等用户发送第一条消息时再创建会话。
- 删除会话：必须二次确认。
- 未登录本地保存：localStorage 最多保留最近 10 个会话。
- 登录用户云端保存失败：自动降级保存到 localStorage，并提示“已临时保存在本机，未同步云端”。
- 降级提示：失败当下 toast 提示，历史栏中持续显示“未同步”标识。
- 登录后显示：历史栏同时显示云端会话和本机会话，统一按排序规则排列。
- 排序规则：置顶优先；置顶内部按更新时间倒序；普通会话按更新时间倒序。
- 置顶能力：第一版支持置顶/取消置顶，最多置顶 4 个会话；只显示小标识，不显示“置顶/最近”分区标题。
- 搜索能力：第一版支持按标题和消息内容搜索；搜索框放在“新对话”按钮下方；placeholder 为“搜索历史”。
- 搜索范围：云端消息内容只搜索最近 50 个云端会话；本地只搜索最多 10 个本机会话。
- 搜索防抖：本地搜索立即执行，云端搜索输入停顿 300ms 后执行。
- 列表数量：默认加载最近 50 个云端会话，不做“加载更多”。
- 空状态：没有任何会话时列表区域保持空白；搜索无结果时显示“没有找到相关历史”。
- 移动端：历史栏默认隐藏，通过按钮打开抽屉，选择会话后关闭。
- 会话操作入口：每个会话 hover 时显示更多按钮，菜单包含重命名、删除、置顶/取消置顶；移动端更多按钮可常驻。
- 重命名交互：使用项目内弹窗重命名。
- 删除确认：使用项目内自定义确认弹窗。
- 模型上下文：每次请求只带当前会话最近 20 条消息。
- assistant 回复保存：流式结束后一次性保存完整回复，不保存半截回复。
- assistant 来源提示：保存当次 `contextSources` 来源摘要，不保存完整日历或日记上下文。
- 单条消息删除：第一版不支持，只支持删除整个会话。
- 数据库变更：允许新增 `chat_sessions` 和 `chat_messages` 两张表，SQL 放在 `supabase/migrations/`，按时间戳命名。
- 文档同步：独立保留本计划，并同步更新 `个人时间Agent完整版计划.md` 路线图。

### 3.1 本版本要做

- 保存聊天会话。
- 保存用户消息和 assistant 回复。
- 支持加载最近一个会话。
- 支持新建会话。
- 支持删除会话。
- 支持手动重命名会话标题。
- 支持置顶最多 4 个会话。
- 支持按标题和消息内容搜索历史会话。
- 支持在当前会话中继续对话。
- 请求模型时携带当前会话最近 20 条消息。
- 登录用户优先保存到 Supabase。
- 未登录用户可使用 localStorage 作为临时本地会话。

### 3.2 本版本不做

- 不做长期记忆。
- 不做用户画像。
- 不做自动偏好提取。
- 不做向量检索。
- 不做聊天内容全文搜索。
- 不做跨会话语义召回。
- 不做自动改日程。
- 不做多 Agent 编排。
- 不保存完整日历或日记上下文快照，只保存必要的来源提示摘要。

## 4. 数据设计

### 4.1 chat_sessions

用于保存一次聊天会话。

字段建议：

```sql
id uuid primary key
user_id uuid not null
title text not null
pinned boolean not null default false
created_at timestamptz not null default now()
updated_at timestamptz not null default now()
```

说明：

- `title` 第一版可以用用户第一条消息前 20 个字生成。
- 后续可以由模型自动总结标题。
- `pinned` 用于置顶排序，第一版最多允许用户置顶 4 个会话。
- `updated_at` 用于会话列表排序。

### 4.2 chat_messages

用于保存会话中的每条消息。

字段建议：

```sql
id uuid primary key
session_id uuid not null
user_id uuid not null
role text not null
content text not null
context_sources jsonb
created_at timestamptz not null default now()
```

说明：

- `role` 只允许 `user` 或 `assistant`。
- `context_sources` 只保存来源标签、缺失来源和可信度，不保存完整日记原文或完整日历上下文。
- 消息内容属于隐私数据，必须受 RLS 保护。
- 第一版不支持删除单条消息，只支持删除整个会话。

## 5. RLS 边界

如果使用 Supabase，需要新增 `chat_sessions` 和 `chat_messages` 表，并开启 RLS。

基础策略：

- 用户只能 select 自己的会话。
- 用户只能 insert 自己的会话。
- 用户只能 update 自己的会话标题和更新时间。
- 用户只能 update 自己的会话置顶状态。
- 用户只能 delete 自己的会话。
- 用户只能 select 自己会话下的消息。
- 用户只能 insert 自己会话下的消息。
- 用户只能 delete 自己会话下的消息。

风险：

- 这是数据库 schema 变更，实施前需要单独确认。
- 不应修改现有 `events`、`tags`、`diary_entries` 的 RLS。
- 不应使用 `service_role` 绕过客户端 RLS。

## 6. 前端交互设计

### 6.1 聊天首页

如果没有历史会话：

- 展示当前 V1 的初始态。
- 用户发送第一条消息时自动创建新会话。

如果有历史会话：

- 默认加载最近一次会话。
- 聊天消息恢复到页面中。
- 用户可以继续输入。

### 6.2 会话列表

第一版做完整历史会话栏：

- 会话栏放在主导航栏右侧，聊天主内容区左侧。
- 会话栏宽度约 260px。
- 顶部显示“新对话”按钮。
- “新对话”按钮下面显示搜索框，placeholder 为“搜索历史”。
- 列表中显示会话标题和更新时间。
- 会话标题单行省略号截断，hover 可显示完整标题。
- 更新时间使用相对时间格式。
- 当前选中的会话有高亮状态。
- 每个会话 hover 时显示更多按钮，菜单提供重命名、删除、置顶/取消置顶。
- 会话标题使用第一条用户消息生成。
- 置顶会话排在最前，最多置顶 4 个；置顶和普通会话内部均按更新时间倒序。
- 置顶会话只显示小标识，不显示“置顶/最近”分区标题。
- 登录后同时显示云端会话和本机会话，本机会话显示“本机”，云端失败降级的会话显示“未同步”。
- 默认加载最近 50 个云端会话，不做加载更多。
- 没有任何会话时列表保持空白；搜索无结果时显示“没有找到相关历史”。

移动端：

- 历史栏默认隐藏。
- 聊天主区域顶部提供历史按钮。
- 点击后打开历史抽屉。
- 选择会话后自动关闭抽屉。

点击“新对话”时：

- 先进入空白聊天页。
- 清空当前 `messages`。
- 设置 `activeSessionId = null`。
- 等用户发送第一条消息时再创建新会话。
- 如果用户没有发送就离开，不保存空会话。

删除会话时：

- 必须二次确认。
- 确认后删除会话及其消息。
- 如果删除的是当前会话，自动切换到最近一个会话。
- 如果没有其他会话，回到空白新对话状态。

重命名会话时：

- 点击更多菜单中的“重命名”后，打开项目内弹窗。
- 标题为空时回退为“新对话”。
- 保存失败时保留原标题并提示失败。

置顶会话时：

- 点击更多菜单中的“置顶”或“取消置顶”。
- 最多允许 4 个置顶会话。
- 如果超过上限，提示“最多置顶 4 个会话”。

搜索历史时：

- 本地搜索立即执行。
- 云端搜索输入停顿 300ms 后执行。
- 搜索范围包含会话标题和消息内容。
- 云端消息内容只在最近 50 个云端会话内搜索。
- 搜索结果仍然返回会话，不返回单条消息。
- 搜索结果排序仍然置顶优先，再按更新时间倒序。

### 6.3 发送流程

用户发送消息时：

1. 如果当前没有 `sessionId`，先创建会话。
2. 保存 user message。
3. 构建日历和日记上下文。
4. 将当前会话最近 20 条消息和最新用户问题发送给 `/api/chat`。
5. 流式接收 assistant 回复。
6. 回复完成后保存 assistant message。
7. 更新会话 `updated_at`。

失败处理：

- 如果 user message 保存失败，不发送给模型。
- 如果 assistant 流式回复失败，页面保留错误状态，不保存空回复。
- 如果 assistant 已有部分回复但中途失败，第一版不保存半截回复。
- assistant 回复只在流式结束且内容非空后一次性保存。
- 如果登录用户云端保存失败，自动降级保存到 localStorage，并提示“已临时保存在本机，未同步云端”。
- 降级发生时，当下显示 toast，历史栏持续显示“未同步”标识。
- 降级保存不能静默伪装成云端同步成功。

## 7. 上下文拼接策略

每次请求模型时，推荐顺序：

```text
系统提示词
当前会话最近 20 条消息
本次日历上下文
本次日记上下文
用户最新问题
```

注意：

- 当前会话消息用于理解多轮对话。
- 日历和日记上下文仍然由 `buildAgentClientContext` 动态生成。
- 不把历史所有会话都传给模型。
- 不把完整日记历史或完整日历明细塞进聊天记录。

## 8. 本地降级策略

Supabase 不可用或用户未登录时：

- 使用 localStorage 保存临时聊天会话。
- localStorage 会话只在本设备可用。
- 未登录状态最多保留最近 10 个会话。
- 超过 10 个时，按 `updated_at` 删除最旧会话和对应消息。
- UI 应标记为“本地会话”或至少不承诺跨设备同步。
- 用户登录后，历史栏仍同时显示本机会话和云端会话。
- 第一版不自动迁移未登录时的本机会话到 Supabase。
- 登录状态新建会话默认创建云端会话，云端失败后再降级本地保存。

建议本地 key：

```text
timestep-chat-sessions-v1
timestep-chat-messages-v1
```

如果后续登录成功，可以再考虑是否提供“迁移本地会话到云端”，本版本不做。

## 9. 隐私和安全要求

- 不在日志中打印聊天正文。
- 不在错误信息中展示完整请求体。
- 不保存 API Key、JWT、请求头或环境变量。
- `context_sources` 只保存来源摘要，不保存完整上下文。
- 删除会话时应同时删除该会话消息。
- 数据库策略必须确保用户只能访问自己的聊天数据。
- 搜索只在当前用户自己的会话和消息内执行。
