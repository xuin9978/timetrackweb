# 个人时间 Agent 聊天功能 PRD V1

## 1. 一句话定位

在现有日历、日记、历史记录基础上，新增一个“个人时间 Agent”聊天页面，让用户可以用自然语言询问今天、本周和近期状态，并获得基于日程与日记上下文的计划建议、复盘总结和时间安排草案。

第一版目标不是做一个通用 ChatGPT，而是做一个能理解用户时间上下文的私人计划助手。

## 2. 背景与机会

当前产品已经具备个人时间管理的核心数据：

- 日历事件：用户每天做了什么、持续多久、属于什么标签。
- 标签系统：用户如何分类自己的工作、生活、学习和休息。
- 历史记录：某一天或某个时间段的实际记录。
- 日记模块：用户对当天状态、想法、复盘和情绪的自然语言记录。

这些数据目前主要用于展示和统计，还没有形成自然语言层面的解释、建议和记忆。因此新增聊天 Agent 的机会是：让用户不用手动翻日历和日记，也能快速知道“我现在该做什么”“我最近为什么累”“这周应该怎么安排”。

## 3. 目标用户

第一版主要服务以下用户：

- 已经在产品里记录日程、时间块或日记的个人用户。
- 希望每天开始前快速生成今日安排的人。
- 希望基于真实记录做轻量复盘的人。
- 希望 AI 能理解自己近期状态，但又不希望 AI 自动修改日程的人。

第一版暂不服务：

- 团队协作日历用户。
- 需要多人会议协调的人。
- 需要复杂目标管理、项目管理、自动排期的人。
- 希望 AI 自动接管所有日程的人。

## 4. 用户问题

### 核心问题

用户现在有日程和日记，但这些数据分散在不同页面里。用户想知道：

- 今天应该优先做什么？
- 本周安排是否过载？
- 最近哪些时间段最适合深度工作？
- 日记里反复出现的问题和日程有什么关系？
- 我接下来应该如何调整自己的时间？

### 第一版要解决的问题

- 用户可以在聊天页面直接提问。
- Agent 能自动带入分层日历上下文：优先参考今天和本周，其次参考本月趋势，同时覆盖全部已加载日历时间段的历史摘要。
- Agent 能自动带入分层日记上下文：优先参考最近 7 天日记，其次参考本月日记趋势，同时覆盖全部日记的长期摘要。
- Agent 能在同一个回答中同时使用日历、日记和上下文来源提示，形成完整的“时间记录 -> 状态理解 -> 行动建议”闭环。
- Agent 能用中文给出简洁、可执行的计划建议。
- Agent 回答时说明参考了哪些上下文类型。

### 第一版不解决的问题

- 不自动修改日程。
- 不做长期主动提醒。
- 不做复杂多 Agent。
- 不做全量历史向量检索。
- 不做跨设备加密记忆同步。
- 不把所有日记原文或所有日历明细无差别直接塞进模型上下文。

## 5. 产品原则

### 5.1 基于真实数据回答

Agent 必须优先基于用户的日历、日记和当前时间回答。没有数据时应明确说明，而不是假装了解用户。

### 5.2 先建议，后行动

第一版只生成建议和草案。任何写入日历、修改事件、删除记录、保存长期记忆的行为都不在 V1 自动执行。

### 5.3 上下文透明

用户需要知道本次回答参考了什么，例如：

- 已参考：今日日程
- 已参考：本周日程
- 已参考：本月时间趋势
- 已参考：最近 7 天日记
- 已参考：全部日记长期摘要
- 未参考：长期记忆

### 5.4 记忆谨慎写入

第一版可以展示“可能值得记住的偏好”，但不自动写入长期画像。长期记忆需要后续版本增加确认和管理入口。

## 6. 第一版功能范围

## 6.1 聊天入口

### 功能描述

在左侧侧边栏“日记”下方增加“聊天”入口。点击后进入聊天页面。

### 用户故事

作为用户，我希望在日记下方看到聊天入口，这样我可以从时间记录和日记自然进入 AI 对话。

### 验收标准

- 侧边栏出现“聊天”按钮。
- 按钮样式与日历、历史、闹钟、日记保持一致。
- 点击后右侧主区域切换为聊天页面。
- 当前选中状态有明确视觉反馈。

## 6.2 ChatGPT 风格聊天首页

### 功能描述

聊天页面初始态展示一个居中的提示语和输入框，整体视觉接近 ChatGPT 的简洁输入体验，同时保持当前应用的浅色、圆角、克制风格。

### 初始态内容

- 主标题：今天有什么计划？
- 输入框 placeholder：问问你的时间 Agent
- 快捷问题：
  - 帮我安排今天的优先级
  - 复盘最近状态和长期模式
  - 看看这周有没有过载

### 验收标准

- 页面没有营销式介绍。
- 用户进入页面后可以直接输入问题。
- 输入框支持 Enter 发送。
- 发送中有 loading 状态。
- 失败时显示可理解的错误信息。

## 6.3 日历上下文注入

### 功能描述

用户发送消息时，前端或后端自动整理用户已有日历时间段，并作为结构化上下文传给 LLM。日历上下文采用“全量覆盖，近期优先”的策略：优先提供今天和本周的详细事件，其次提供本月趋势，最后提供全部历史时间段的摘要画像。

### 第一版上下文范围

- 今天的事件：提供完整事件明细。
- 本周的事件：提供完整事件明细，用于判断过载、空档和优先级。
- 本月趋势：提供按日期、标签、总时长、忙碌程度聚合后的统计。
- 全部历史摘要：覆盖所有已加载日历时间段，但只提供长期统计画像，不直接传入全部事件明细。
- 近期详细事件包含：
  - 标题
  - 开始时间
  - 结束时间
  - 持续时长
  - 标签名称
- 历史摘要包含：
  - 总记录天数
  - 总事件数
  - 高频标签
  - 常见忙碌时段
  - 常见空档时段
  - 长期过载模式，如果可计算

### 示例上下文结构

```json
{
  "calendarContext": {
    "date": "2026-07-18",
    "todayEvents": [
      {
        "title": "写 PRD",
        "start": "09:30",
        "end": "11:00",
        "durationMinutes": 90,
        "tag": "工作"
      }
    ],
    "weekSummary": {
      "eventCount": 18,
      "busyDays": ["Monday", "Wednesday"],
      "totalScheduledMinutes": 1260
    },
    "monthSummary": {
      "eventCount": 64,
      "totalScheduledMinutes": 4280,
      "topTags": ["工作", "学习", "运动"],
      "overloadedDays": ["2026-07-10", "2026-07-16"]
    },
    "allTimeSummary": {
      "eventCount": 312,
      "trackedDays": 86,
      "topTags": ["工作", "学习"],
      "commonBusyWindows": ["09:00-12:00", "20:00-22:00"],
      "commonFreeWindows": ["14:00-16:00"]
    }
  }
}
```

### 验收标准

- 用户问“今天怎么安排”时，回答能引用今天的真实事件。
- 用户问“这周忙不忙”时，回答能结合本周事件数量和时间分布。
- 用户问“我长期的时间使用有什么问题”时，回答能基于全部历史摘要给出趋势判断。
- 没有日程数据时，Agent 明确说明“当前没有可参考日程”。

## 6.4 日记上下文注入

### 功能描述

第一版接入用户已有日记上下文，并采用“全量覆盖，近期优先”的策略：优先参考最近 7 天日记的较详细内容，其次参考本月日记趋势，最后参考全部日记的长期摘要。日记上下文优先使用摘要、主题和情绪线索，不建议直接传入大量原文。

### 第一版上下文范围

- 最近 7 天日记：提供较详细的摘要和必要片段。
- 本月日记趋势：提供主题、情绪、重复关键词和高频问题。
- 全部日记长期摘要：覆盖所有日记，但只提供长期主题画像，不传入全部原文。
- 最近 7 天每篇日记可传入：
  - 日期
  - 标题或首句
  - 简短摘要
  - 情绪或主题标签，如果已有
- 本月和全部历史摘要可传入：
  - 日记数量
  - 高频主题
  - 重复出现的问题
  - 情绪变化趋势
  - 与日历时间分布可能相关的线索

如果当前没有日记摘要能力，最近 7 天日记可以先在请求时截取每篇日记的前 300 字；本月和全部历史必须先做聚合或摘要，不应直接把全部日记原文塞进模型上下文。后续版本再改为摘要表、向量检索或长期记忆表。

### 验收标准

- 用户问“最近状态怎么样”时，Agent 能结合最近日记回答。
- 用户问“我长期反复出现的问题是什么”时，Agent 能基于全部日记摘要回答。
- Agent 不直接暴露完整日记原文，除非用户明确要求查看。
- 没有日记时，Agent 不编造用户状态。

## 6.5 LLM 回复能力

### 功能描述

聊天请求通过后端接口调用 LLM API。前端不直接暴露 API Key。

### 第一版模型能力

Agent 应支持以下问题：

- 今日计划：根据今天日程给出优先级建议。
- 本周复盘：根据本周日历、最近 7 天日记和本月趋势总结状态。
- 长期复盘：根据全部日历摘要和全部日记长期摘要识别长期时间模式。
- 过载识别：判断某天是否安排过密。
- 空档建议：识别可用于休息或深度工作的空白时间段。
- 轻量陪伴：对用户状态给出克制、具体的建议。

### 系统提示词方向

Agent 的身份应该是：

> 你是一个个人时间管理 Agent。你会基于用户的分层日历上下文、分层日记上下文和当前日期回答问题。你必须优先参考最近 7 天和本周的详细上下文，再参考本月趋势，最后参考全部历史摘要。你必须诚实说明哪些信息来自上下文，哪些是推断。不要假装知道未提供的数据。涉及修改日程时，只能提出草案，不能声称已经完成修改。

### 验收标准

- API Key 只从服务端环境变量读取。
- 前端请求 `/api/chat`，不携带密钥。
- 模型返回中文。
- 模型回答简洁，有行动建议。
- 错误时不泄露 API Key、请求头或内部堆栈。

## 6.6 上下文来源提示

### 功能描述

每次回答后，页面展示本次使用了哪些上下文。

### 展示示例

```text
已参考：今日日程、本周日程、本月时间趋势、最近 7 天日记、全部日记长期摘要
未参考：长期记忆
```

### 验收标准

- 用户能知道回答是否基于真实数据。
- 无数据时显示“未找到可参考数据”。
- 不展示敏感原文和密钥。

## 8. 技术实现方案

## 8.1 前端

建议新增或维护以下模块：

- `components/Chat.tsx`：聊天页面 UI 和状态。
- `utils/chatService.ts`：前端请求 `/api/chat`。
- `App.tsx`：将 `events`、`tags`、当前日期传给 `Chat`。

前端发送内容：

```ts
{
  messages: ChatMessage[],
  quality: "fast" | "balanced" | "high",
  clientContext: {
    currentDate,
    calendarContext: {
      todayEvents,
      weekEvents,
      monthSummary,
      allTimeSummary
    },
    diaryContext: {
      recentSevenDays,
      monthSummary,
      allTimeSummary
    }
  }
}
```

## 8.2 后端接口

建议使用：

- `api/chat.ts`：部署环境的 serverless API。
- `server/llmChat.ts`：LLM 请求封装。
- `vite.config.ts`：本地开发时模拟 `/api/chat`。

后端职责：

- 校验请求体。
- 读取环境变量里的 API Key。
- 拼装系统提示词。
- 拼装结构化上下文。
- 调用 LLM。
- 返回回答和上下文来源。

### 后端配置与调用约定

后端需要记录的是配置项名称、职责和调用链，不记录任何真实密钥值。

建议环境变量：

- `DEEPSEEK_API_KEY`：DeepSeek API Key，仅服务端读取。
- `DEEPSEEK_API_BASE_URL`：DeepSeek 接口地址，默认可使用 `https://api.deepseek.com`。
- `DEEPSEEK_MODEL`：默认模型，例如 `deepseek-chat`。
- `LLM_API_KEY`：通用 LLM API Key 备用字段。
- `LLM_API_BASE_URL`：通用 LLM 接口地址备用字段。
- `LLM_MODEL`：通用 LLM 模型备用字段。

后端调用链：

```text
Chat.tsx
  -> utils/chatService.ts
  -> POST /api/chat
  -> api/chat.ts
  -> server/llmChat.ts
  -> LLM chat completions
  -> 流式返回给前端
```

上下文记忆的调用原则：

- 每次用户发送消息时，都应携带本次可用的 `clientContext`。
- 后端负责将 `clientContext` 转换为稳定的 `User Data Context`，再与系统提示词、用户消息一起传给 LLM。
- 后端应在模型输入中保留“上下文来源摘要”，让模型知道哪些数据可用、哪些数据不可用。
- 后端返回结果时，应同时返回或支持前端展示 `contextSources`，用于说明本次回答参考了哪些上下文。
- V1 的聊天记录可以只保存在前端状态中；如果需要在刷新后保留回答和上下文来源，则进入 V1.5 的聊天记录表。

建议返回结构：

```ts
{
  message: string,
  model: string,
  contextSources: {
    used: string[],
    missing: string[],
    confidence: "high" | "medium" | "low"
  }
}
```

流式响应场景下，正文可以通过 SSE 分片返回；`contextSources` 可以在响应开始前由前端根据本次请求上下文展示，也可以在后续版本通过独立元信息事件返回。

## 8.3 数据读取策略

第一版优先采用前端已有状态传入上下文，减少 Supabase 服务端鉴权复杂度。

原因：

- 当前应用主要在浏览器端用 Supabase anon key 和用户 session 管理数据。
- 服务端直接读用户数据需要处理 Supabase JWT、RLS 和鉴权边界。
- MVP 阶段更适合先复用前端已加载事件，验证聊天体验是否成立。

后续版本再升级为服务端鉴权读取：

```text
前端传 Supabase access token
  -> 后端校验用户身份
  -> 后端按用户查询 events / diary_entries / memories
  -> 构建上下文
  -> 调用 LLM
```

## 9. 数据模型建议

## 9.1 V1 可不新增数据库表

第一版可以不新增表，只使用现有 events、tags、diary 数据，并将聊天记录保存在前端状态中。

## 9.2 V1.5 可新增聊天记录表

```sql
agent_chat_sessions
- id
- user_id
- title
- created_at
- updated_at

agent_chat_messages
- id
- session_id
- user_id
- role
- content
- context_summary
- context_sources
- created_at
```

其中：

- `content` 保存用户问题或 Agent 回答。
- `context_summary` 保存本次调用时使用的上下文摘要，例如“参考了今日日程、本周日程、本月趋势、最近 7 天日记”。
- `context_sources` 保存结构化来源列表，例如 `used` 和 `missing`，用于回看时还原回答依据。
- 不建议保存完整日记原文、完整请求体或任何 API Key。

## 9.3 V2 可新增记忆表

```sql
agent_memories
- id
- user_id
- type
- content
- source_type
- source_ids
- confidence
- status
- created_at
- updated_at
- last_used_at
```

`status` 可取：

- `candidate`：候选记忆，等待用户确认。
- `active`：已启用。
- `dismissed`：用户拒绝。
- `archived`：已归档。

## 10. Prompt 结构

第一版建议固定为四层：

```text
1. Stable System Prompt
   你是谁、能做什么、不能做什么、安全边界。

2. Dynamic Time Context
   当前日期、星期、用户所在时区。

3. User Data Context
   分层用户数据上下文：
   - 近期详细上下文：今日日程、本周日程、最近 7 天日记。
   - 中期趋势上下文：本月时间分布、本月日记主题趋势。
   - 全量历史摘要：全部日历时间段摘要、全部日记长期摘要。

4. Conversation Messages
   用户当前问题和最近聊天上下文。
```

注意：稳定系统提示词不要频繁变化，动态上下文放后面。

## 11. 权限边界

第一版只允许：

- 读取上下文。
- 总结上下文。
- 给出建议。
- 生成日程草案。

第一版不允许：

- 自动创建日程。
- 自动修改日程。
- 自动删除日程。
- 自动保存长期记忆。
- 自动发送外部消息。

当 Agent 生成可执行建议时，文案应为：

```text
我可以帮你生成一个日程草案。当前版本不会自动写入日历。
```

## 12. 关键用户流程

## 12.1 今日计划

```text
用户进入聊天页
  -> 点击“帮我安排今天的优先级”
  -> 系统读取今日日程、本周摘要、本月时间趋势和历史时间摘要
  -> Agent 输出：
     1. 今日已有安排
     2. 推荐优先级
     3. 可用空档
     4. 一个具体执行建议
```

## 12.2 周复盘

```text
用户输入“复盘一下我这周”
  -> 系统读取本周日程、最近 7 天日记、本月趋势和全部历史摘要
  -> Agent 输出：
     1. 本周时间分布
     2. 日记中反复出现的主题
     3. 可能的压力来源
     4. 下周调整建议
```

## 12.3 过载判断

```text
用户输入“我明天是不是太满了”
  -> 系统识别时间范围为明天
  -> 读取明天事件
  -> 计算总安排时长、连续事件、空档
  -> Agent 输出是否过载和调整建议
```

## 13. 成功指标

### 产品指标

- 用户每周至少打开聊天页 2 次。
- 用户发送后得到可用回答的比例超过 70%。
- 快捷问题点击率超过 30%。
- 用户连续 2 周使用“今日计划”或“周复盘”。

### 质量指标

- 回答中出现虚构日程的比例接近 0。
- 缺少上下文时能明确说明。
- LLM 错误不暴露敏感信息。
- 回答平均延迟可接受，第一版目标小于 10 秒。

## 14. 风险与对策

### 风险 1：Agent 看起来像普通聊天机器人

对策：每次回答都展示参考了哪些日程和日记；快捷问题围绕时间管理，而不是泛聊天。

### 风险 2：模型编造用户状态

对策：系统提示词要求区分事实和推断；没有上下文时必须说明。

### 风险 3：日记隐私敏感

对策：第一版优先传摘要、主题和必要片段，不传全量原文；最近 7 天可以保留更多细节，本月和全部历史只传聚合摘要；不在日志中保存完整请求体。

### 风险 4：全量上下文过长

对策：采用分层上下文。近期数据提供明细，中期数据提供趋势，全部历史提供摘要；后续可升级为摘要表、向量检索和长期记忆。

### 风险 5：服务端鉴权复杂

对策：第一版先由前端传当前已加载上下文；第二版再做服务端 Supabase JWT 鉴权。

### 风险 6：长期记忆误写

对策：第一版不自动写长期记忆；后续新增候选记忆和用户确认机制。

### 风险 7：完整方案首版范围过大

对策：完整方案只覆盖“读上下文、总结上下文、展示来源、给出建议”，不扩展到自动写入日程、长期记忆、聊天持久化和向量检索。开发时按日历上下文、日记上下文、后端 Prompt、前端来源提示、验证截图五个步骤推进，避免同时改动过多行为。

## 15. 完整方案实施计划

本轮 V1 采用完整方案：日历上下文、日记上下文、上下文来源提示一起完成。目标是让聊天功能从“通用 LLM 对话”升级为“可解释的个人时间 Agent”。

### 15.1 实施边界

本轮必须完成：

- 接入日历分层上下文。
- 接入日记分层上下文。
- 后端 Prompt 使用日历和日记上下文。
- 前端展示上下文来源提示。
- 空数据、无权限、LLM 错误时有明确降级反馈。

本轮不做：

- 自动创建、修改、删除日程。
- 自动保存长期记忆。
- 聊天记录数据库持久化。
- 向量检索或 RAG。
- 服务端 Supabase 鉴权读取用户数据。

### 15.2 实施顺序

1. 日历上下文构造
   - 从当前已加载 `events` 和 `tags` 中构造 `todayEvents`、`weekEvents`、`monthSummary`、`allTimeSummary`。
   - 今天和本周保留事件明细，本月和全部历史只传统计摘要。

2. 日记上下文构造
   - 从当前用户日记数据中构造 `recentSevenDays`、`monthSummary`、`allTimeSummary`。
   - 最近 7 天可传摘要和必要片段，本月和全部历史只传主题、情绪和重复问题摘要。
   - 如果当前没有摘要字段，最近 7 天可临时截取前 300 字；历史部分必须聚合后再传。

3. 请求协议升级
   - `Chat.tsx` 发送 `messages`、`quality` 和 `clientContext`。
   - `utils/chatService.ts` 透传 `clientContext` 到 `/api/chat`。
   - `server/llmChat.ts` 校验并压缩上下文，生成稳定的 `User Data Context`。

4. 后端 Prompt 升级
   - Prompt 必须要求模型区分事实、统计和推断。
   - Prompt 必须要求模型在缺少上下文时明确说明。
   - Prompt 必须禁止模型声称已经修改日程或保存记忆。

5. 上下文来源提示
   - 前端在 Agent 回答旁展示本次 `contextSources`。
   - 至少区分 `used`、`missing` 和 `confidence`。
   - 流式响应场景下，V1 可以先由前端根据本次请求上下文生成来源提示；后续再由后端流式返回元信息。

6. 验证与作品集素材
   - 跑 `npm run build`。
   - 准备三类演示问题：今日计划、周复盘、长期模式。
   - 截图保留：首页、流式回答、上下文来源提示、无数据或缺权限提示。

### 15.3 完整方案验收问题

V1 至少能回答：

- “帮我安排今天的优先级”：基于今日日程、本周安排和历史时间摘要回答。
- “复盘最近状态和长期模式”：基于最近 7 天日记、本月趋势和全部历史摘要回答。
- “看看这周有没有过载”：基于本周日历明细、本月趋势和过载规则回答。
- “我长期的时间使用有什么问题”：基于全部日历和日记摘要回答，并说明这是趋势判断而不是确定结论。

## 16. 版本迭代规划

## V1：上下文聊天 MVP

包含：
