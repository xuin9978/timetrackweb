# 个人时间 Agent 完整版计划

## 1. 当前状态

当前已经完成的是个人时间 Agent V1。它已经具备聊天入口、基础聊天界面、流式回复、日历上下文、日记上下文、来源提示和服务端 LLM 调用。

V1 的本质是“带时间上下文的聊天助手”：用户提问时，前端整理已有日历和日记数据，后端把这些上下文注入模型提示词，模型再给出计划建议或复盘回答。

目前已经完成：

- 聊天入口：侧边栏已加入“聊天”模块。
- 聊天 UI：输入框、快捷问题、loading、错误提示、流式输出。
- 日历上下文：今日日程、本周日程、本月时间趋势、全部历史时间摘要。
- 日记上下文：最近 7 天日记、本月日记趋势、全部日记长期摘要。
- 来源提示：展示已参考、未参考、可信度，且默认折叠。
- 服务端调用：通过 `/api/chat` 调用 LLM，前端不暴露 API Key。
- 安全边界：不自动修改日程，不保存长期记忆，不直接传完整日记历史。
- 同步保障：已修复 Supabase Auth metadata 过大导致时间段保存失败的问题，并沉淀 guardrail。

当前还没有完成：

- 对话自然度还不够，回答偏分析报告。
- 没有聊天记录持久化。
- 没有长期记忆和用户画像。
- 没有 Agent 工具调用能力。
- 没有确认后写入日程。
- 没有主动提醒和周期复盘。
- 没有 RAG 或语义检索。
- 没有上下文调试面板。
- 没有完整的作品集叙事包装。

## 2. 完整版目标

完整版个人时间 Agent 的目标不是“在日历里加一个 ChatGPT”，而是让产品具备一个真正理解用户时间、状态和目标的个人 Agent。

完整版应形成下面的能力闭环：

用户记录时间和日记。

Agent 理解用户近期状态、长期模式和时间分布。

Agent 用自然对话方式给出计划、复盘和提醒。

用户确认后，Agent 可以生成日程草案并写入日历。

系统沉淀用户偏好、节奏、长期目标和反复问题。

用户可以查看、修改、删除这些记忆和 Agent 判断依据。

## 3. 产品原则

一、自然，但不装熟

Agent 要像一个熟悉用户节奏的时间伙伴，而不是报告生成器。它可以温和、具体、有陪伴感，但不能假装知道没有被记录的信息。

二、基于真实上下文

凡是涉及日程、状态、长期模式的判断，都必须来自日历、日记、聊天历史、用户确认过的记忆或明确推断。

三、先建议，后确认，再行动

任何写入日历、修改事件、删除数据、保存长期记忆的动作，都必须经过用户确认。

四、上下文透明

用户应该知道 Agent 参考了什么，也应该能看到它为什么这样判断。

五、记忆可控

长期记忆必须可查看、可编辑、可删除、可关闭。

六、隐私优先

日记、聊天、用户画像都属于高隐私数据。默认不暴露、不打印、不无差别传给模型。

## 4. 能力分层

### 4.1 L1：上下文聊天 Agent

定位：回答用户关于今天、本周、近期状态和长期时间使用的问题。

当前 V1 已基本完成。

还需要补强：

- 优化自然语言风格，减少报告感。
- 根据问题类型切换回答结构。
- 普通问题简短回答，复盘问题再展开分析。
- 避免输出 Markdown 符号。
- 增加对“用户情绪”和“任务压力”的轻量回应。

### 4.2 L2：可解释 Agent

定位：用户不仅能得到回答，还能知道回答依据。

需要新增：

- 回答依据面板。
- 本次使用的日程片段。
- 本次使用的日记摘要。
- 本次缺失的数据。
- 判断可信度。
- 哪些内容是事实，哪些是推断。

V1 已经有来源提示，但还不够细。完整版需要从“来源标签”升级为“可展开依据”。

### 4.3 L3：多轮记忆 Agent

定位：Agent 能记住同一轮或跨轮对话里的任务、偏好和上下文。

需要新增：

- 聊天记录持久化。
- 会话列表。
- 当前会话上下文压缩。
- 用户可清空会话。
- 重要偏好的候选记忆。
- 用户确认后写入长期记忆。

第一阶段只保存聊天记录，不自动生成长期记忆。

第二阶段再加入“可能值得记住”的提示。

第三阶段加入用户确认后的长期记忆。

### 4.4 L4：用户画像 Agent

定位：Agent 能形成用户的长期时间画像。

画像内容可以包括：

- 常见高效时间段。
- 常见过载时间段。
- 高频任务类型。
- 长期拖延主题。
- 情绪波动和日程之间的关系。
- 面试、作品集、简历等阶段性重点。
- 用户偏好的计划风格，例如简短、温和、严格、陪伴式。

用户画像不能自动强写。每条画像都要有来源、置信度和编辑入口。

### 4.5 L5：工具调用 Agent

定位：模型不只是被动接收上下文，而是可以按需调用工具。

可提供的工具：

- 查询某天日程。
- 查询某周日程。
- 查询某月统计。
- 查询最近日记。
- 搜索历史日记。
- 生成日程草案。
- 检查某个时间段是否冲突。
- 保存用户确认后的日程。

V1 是“前端一次性打包上下文”。完整版应该逐步升级为“Agent 按需查询工具”。

### 4.6 L6：行动 Agent

定位：用户确认后，Agent 可以执行轻量操作。

可执行动作：

- 生成今天计划草案。
- 把计划草案转为日历事件。
- 调整某个时间段。
- 将复盘结论保存为日记。
- 将用户确认的偏好保存为记忆。
- 生成下一周安排建议。

所有行动必须走确认流程：

Agent 提出草案。

用户查看并修改。

用户点击确认。

系统写入数据。

写入后给出结果和回滚入口。

### 4.7 L7：主动 Agent

定位：Agent 可以在合适时机主动提醒和复盘。

可做场景：

- 今天日程过载提醒。
- 长时间没有记录日记提醒。
- 一周复盘提醒。
- 面试或项目截止前的准备提醒。
- 连续晚睡或夜间高强度任务提醒。
- 本周作品集进度提醒。

主动能力必须可关闭，并且要控制频率，不能打扰用户。

## 5. 版本路线图

### V1.1：自然对话优化

目标：让 Agent 像一个自然的时间伙伴，而不是报告生成器。

要做：

- 重写系统提示词，加入自然对话风格。
- 根据问题类型区分回答方式。
- 普通计划问题不强制分事实、统计、推断。
- 复盘类问题才展开结构化分析。
- 禁止 Markdown 符号输出。
- 增加回复质量测试。

验收：

- 问“我今天该先做什么”，回答像自然聊天。
- 问“我最近是不是状态不好”，先回应状态，再给依据。
- 回答中不出现 `#`、`-`、`**` 等 Markdown 标记。

### V1.5：聊天体验和依据展示

目标：让用户更清楚 Agent 参考了什么。

要做：

- 优化消息气泡和滚动体验。
- 来源提示升级为可展开依据。
- 展示本次参考的关键日程。
- 展示本次参考的日记摘要。
- 展示缺失数据。
- 增加“重新生成”和“复制回答”。
- 增加空状态示例问题。

验收：

- 用户能看到回答依据。
- 用户能知道没有读取哪些数据。
- 用户可以复制或重新生成回答。

### V1.6：聊天记录持久化和会话记忆

目标：先补齐第二层多轮记忆，让 Agent 支持会话级连续对话。

要做：

- 新增聊天会话表。
- 新增聊天消息表。
- 保存用户消息和助手回答。
- 在聊天模块内部新增历史会话栏，位置在主导航栏右侧，宽度约 260px。
- 会话列表显示标题和更新时间。
- 支持新建会话。
- 支持继续上次对话。
- 支持手动重命名会话标题。
- 支持删除会话。
- 删除会话前二次确认。
- 登录用户保存到 Supabase。
- 未登录用户使用 localStorage fallback，最多保留最近 10 个会话。
- 登录用户云端保存失败时，自动降级保存到 localStorage，并提示未同步云端。
- 每次请求模型时只带当前会话最近 20 条消息。

验收：

- 刷新页面后聊天记录仍在。
- 用户能回到之前的会话继续聊。
- 用户能新建、切换、重命名和删除会话。
- 删除会话后不可再被 Agent 引用。
- 不自动生成长期记忆。
- 不跨会话语义召回。

### V2：当前会话摘要

目标：当单个会话变长时，用摘要压缩上下文，降低 token 成本。

要做：

- 新增当前会话摘要字段。
- 超过消息阈值后生成或更新 `session_summary`。
- 请求模型时使用会话摘要 + 最近 20 条消息。
- 摘要只服务当前会话，不写入长期记忆。

验收：

- 长会话仍能延续主题。
- 请求不会无限携带全部历史消息。
- 摘要内容不包含日记完整原文或敏感请求体。

### V2.5：长期记忆和用户画像

目标：让 Agent 逐步理解用户的长期节奏。

要做：

- 新增记忆候选机制。
- 用户确认后保存长期记忆。
- 新增记忆管理页面。
- 每条记忆保存来源、创建时间、置信度。
- 支持编辑、删除、禁用记忆。
- Prompt 中加入用户确认过的记忆。

验收：

- Agent 不会自动保存未经确认的记忆。
- 用户能看到 Agent 记住了什么。
- 用户删除记忆后，后续回答不再引用。

### V3：Agent 工具调用

目标：让 Agent 从“被动接收上下文”升级为“按需查询数据”。

要做：

- 定义工具协议。
- 提供日历查询工具。
- 提供日记查询工具。
- 提供统计查询工具。
- 提供冲突检测工具。
- 后端统一执行工具，前端不暴露敏感能力。
- 工具结果加入回答依据。

验收：

- 用户问某一天时，Agent 只查询那一天相关数据。
- 用户问长期问题时，Agent 查询摘要或语义检索结果。
- 工具调用失败时，Agent 明确说明无法读取。

### V3.5：日程草案和确认写入

目标：让 Agent 能辅助安排时间，但不越权。

要做：

- Agent 生成日程草案。
- 前端展示草案卡片。
- 用户可以编辑标题、时间、标签。
- 检查时间冲突。
- 用户确认后写入日历。
- 写入失败时回滚并提示。

验收：

- Agent 不能直接声称“我已经帮你安排好了”。
- 必须经过确认后才写入。
- 写入后日历立即可见，刷新后仍存在。

### V4：RAG 和语义检索

目标：让 Agent 能处理大量历史日记和长期记录。

要做：

- 设计日记摘要表。
- 设计事件摘要表。
- 设计向量索引或语义检索方案。
- 对历史日记做分段摘要。
- 查询时优先检索相关片段，而不是全量塞上下文。
- 增加隐私和删除同步机制。

验收：

- 用户问长期模式时，Agent 能找到相关历史片段。
- 删除日记后，对应索引也会失效。
- Prompt 不包含大量无关原文。

### V5：主动复盘和提醒

目标：让 Agent 从被问才答，升级为合适时机主动提醒。

要做：

- 一周复盘生成。
- 过载提醒。
- 空档建议。
- 目标进度提醒。
- 用户可配置提醒频率。
- 用户可关闭主动 Agent。

验收：

- 提醒不打扰用户。
- 每条提醒都有依据。
- 用户可以关闭或调整提醒类型。

## 6. 技术架构演进

### 当前架构

前端整理上下文。

前端请求 `/api/chat`。

后端拼接 Prompt。

LLM 返回流式回答。

前端展示回答和来源提示。

### 完整版架构

前端负责交互、确认和展示。

后端负责模型调用、工具执行、权限校验和上下文压缩。

数据库负责日历、日记、聊天记录、记忆、工具执行日志。

Agent 编排层负责问题分类、工具选择、上下文组装和动作确认。

## 8. Agent 回答策略

### 普通计划问题

回答方式：

先给一句自然回应。

再给 2 到 4 条具体建议。

最后给一个可以马上执行的小动作。

不要一上来写大量分析。

### 复盘问题

回答方式：

先总结整体状态。

再分日历依据、日记依据、推断。

最后给调整建议。

### 情绪或压力问题

回答方式：

先接住用户感受。

再说明依据是否充分。

再给轻量建议。

避免诊断式表达。

### 行动请求

回答方式：

先生成草案。

明确说明尚未写入。

引导用户确认或修改。

确认后才执行。

## 9. 验收清单

### 自然对话

- 用户连续问 3 个问题，Agent 能自然承接上下文。
- 回答不出现 Markdown 符号。
- 回答不会每次都像报告。

### 上下文准确性

- 今天问题引用今天真实日程。
- 本周问题引用本周真实日程。
- 最近状态问题引用最近 7 天日记。
- 长期问题引用长期摘要或检索结果。
- 没有数据时明确说明缺失。

### 记忆安全

- 不自动保存长期记忆。
- 记忆必须用户确认。
- 用户可删除记忆。
- 删除后不再引用。

### 行动安全

- 不自动写入日程。
- 写入前必须展示草案。
- 用户确认后才保存。
- 保存失败要回滚并提示。

### 隐私安全

- 不打印 API Key。
- 不打印 JWT。
- 不传全部日记原文。
- 不把 Base64 或大字段写入 Supabase Auth metadata。

## 10. 推荐执行顺序

第一步：V1.1 自然对话优化。

原因：这是当前用户最容易感知的问题，改动成本低，作品集效果明显。

第二步：V1.5 依据展示和聊天体验。

原因：让 Agent 的可信度更强，也方便展示“不是普通聊天，而是基于个人时间数据”。

第三步：V2 聊天记录。

原因：没有聊天记录，就很难形成真正的多轮 Agent。

第四步：V2.5 用户画像和长期记忆。

原因：这是 Agent 从工具走向个人化的核心。

第五步：V3 工具调用。

原因：工具调用会让 Agent 能力更像真正的 Agent，而不是一次性上下文问答。

第六步：V3.5 确认后写入日程。

原因：这是从建议到行动的关键一步，但必须放在权限和确认机制之后。

第七步：V4 和 V5。

原因：RAG 和主动提醒价值很高，但复杂度也高，应该在基础闭环稳定后再做。

## 11. 作品集表达方式

可以把这个项目包装成：

从时间管理工具到个人时间 Agent 的产品升级。

核心展示结构：

一、问题背景：用户有日历和日记，但缺少自然语言理解和行动建议。

二、V1 方案：基于日历和日记上下文的聊天 Agent。

三、关键设计：分层上下文、来源提示、安全边界、服务端模型调用。

四、问题修复：Supabase metadata 导致同步失败，如何定位并沉淀 guardrail。

五、下一阶段：自然对话、记忆、工具调用、确认后写入、主动复盘。

六、产品价值：从记录工具变成理解用户时间和状态的智能伙伴。
