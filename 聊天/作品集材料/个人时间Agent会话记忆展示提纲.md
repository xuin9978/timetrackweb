# 展示提纲：个人时间 Agent 会话记忆升级

## 1. 面试 3 分钟讲法

我做的是一个个人时间管理产品里的 AI Agent 升级。产品原本已经有日历和日记，第一版 Agent 能读取这些上下文回答今天、本周和近期状态的问题，但它仍然像一次性问答：刷新页面后对话会丢，用户也不能继续上一轮讨论。

所以我这次补齐的是会话级记忆能力。核心不是长期画像，而是先让用户可以查看历史会话、恢复对话、继续追问，并让模型只读取当前会话最近 20 条消息。

实现上我做了 Supabase 云端持久化和 localStorage fallback。登录用户优先保存云端，未登录或云端失败时保存本机，并用“本机”“未同步”提示状态。数据库新增 `chat_sessions` 和 `chat_messages`，并配置 RLS，确保用户只能访问自己的聊天记录。

交互上，我在聊天页左侧新增历史栏，支持新对话、搜索、重命名、删除确认、置顶和移动端抽屉。新对话不会立刻创建空会话，而是等用户发出第一条消息后再创建。

这版的关键边界是：不自动写长期记忆，不自动修改日程，不保存完整日历和日记上下文，只保存回答来源摘要。最终效果是用户刷新页面后可以继续对话，Agent 也能理解上一轮刚刚聊过的内容。

## 2. 简历项目描述

个人时间 Agent 会话记忆升级：基于 React、TypeScript、Supabase 为 TimeTrack 聊天 Agent 设计并实现会话级多轮记忆。新增聊天历史侧边栏、会话搜索、重命名、删除确认、置顶、移动端抽屉；设计 `chat_sessions` / `chat_messages` 表和 RLS policy，支持 Supabase 云端持久化与 localStorage fallback。模型请求仅携带当前会话最近 20 条消息，并保存来源摘要而非完整日历 / 日记上下文，兼顾连续对话、隐私边界和异常降级体验。

### P5 数据与系统设计

数据表：

- `chat_sessions`
- `chat_messages`

存储策略：

- 登录：Supabase。
- 未登录：localStorage。
- 云端失败：localStorage fallback + 未同步提示。

上下文策略：

- 模型请求只带当前会话最近 20 条消息。
- 日历和日记上下文发送时动态构建。
- 只保存 `context_sources`，不保存完整上下文快照。

### P6 安全与边界

重点讲：

- RLS 限制用户只能访问自己的会话和消息。
- `chat_messages` 的写入必须属于自己的 session。
- 收紧 grant，避免超出最小授权。
- 不使用 service_role 绕过 RLS。
- 不保存完整日记原文到聊天历史。

## 5. 面试追问准备

### 为什么不直接做长期记忆？

因为长期记忆涉及偏好提取、用户确认、删除权和隐私管理。第一版直接自动写长期记忆容易越界。我选择先做会话级记忆，让用户能恢复和继续当前对话，再把长期记忆作为下一阶段，用“候选记忆 + 用户确认 + 管理页”来做。

### 为什么需要 localStorage fallback？

时间管理和复盘类对话通常有即时表达价值。如果云端失败就丢失用户刚刚输入的内容，体验会很糟。fallback 能保证内容先留在本机，同时通过“未同步”提示告诉用户真实状态，不伪装成云端成功。

### 为什么模型只带最近 20 条？

这是连续性和成本之间的折中。最近 20 条足够覆盖当前会话里的短期上下文，也避免把全部历史塞给模型造成 token 成本和隐私风险。后续可以用会话摘要进一步压缩长对话。

### 为什么只保存来源摘要？

用户需要知道回答依据，但聊天记录不应该复制完整日历和日记。来源摘要能保留可解释性，同时减少敏感数据重复存储。

### 如果云端和本地都有同一个会话怎么办？

列表会合并云端和本地会话，并优先保留“未同步”状态，避免把降级保存误展示为完全同步。这样用户能知道这段内容还没有真正同步到云端。

## 6. 作品集页可直接使用的结论

这次迭代的重点不是做一个更复杂的 AI，而是把 Agent 变成一个真实可用的产品功能：它能延续对话、恢复历史、处理异常、解释来源，并且在记忆和隐私之间保持清晰边界。
