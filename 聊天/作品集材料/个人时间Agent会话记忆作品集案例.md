# 作品集案例：个人时间 Agent 的会话级记忆升级

## 1. 项目一句话

把 TimeTrack 从“能回答一次问题的时间聊天助手”，升级为“能延续同一段对话、恢复历史会话、同时保留隐私边界的个人时间 Agent”。

## 2. 背景

TimeTrack 已经有日历、日记、历史记录和标签系统。第一版个人时间 Agent 能够读取日历和日记上下文，回答“今天先做什么”“这周是不是太满”“最近状态有什么规律”等问题。

但第一版仍然存在一个明显断点：聊天只存在于当前页面状态里。用户刷新页面或重新进入聊天模块后，上一段对话会丢失；模型也只能理解当前页面内存中的最近消息，无法形成稳定的会话连续性。

这会让 Agent 更像一个临时问答入口，而不是一个可以长期陪用户复盘时间、承接计划、延续讨论的产品能力。

## 3. 核心问题

这次迭代要解决的不是“让模型更聪明”，而是补齐 Agent 产品里的第二层能力：会话级多轮记忆。

具体问题包括：

- 刷新页面后聊天记录丢失。
- 用户无法查看历史会话。
- 用户不能继续上一次对话。
- 模型请求缺少稳定的当前会话历史。
- 聊天记录如果直接保存完整上下文，会带来隐私和数据膨胀风险。

## 4. 产品目标

这版目标是让用户可以在聊天模块中自然地创建、查看、搜索和恢复历史会话，并让模型在同一个会话里理解最近对话。

范围刻意控制在“会话级记忆”，不做长期记忆、用户画像、RAG、工具调用或自动写日程。

## 5. 方案设计

### 5.1 数据层

采用 Supabase 云端持久化 + localStorage fallback。

- 登录用户优先保存到 Supabase。
- 未登录用户保存到 localStorage。
- 登录用户云端保存失败时，降级保存到本机，并显示“未同步”。
- 本地最多保留最近 10 个会话。
- 云端默认加载最近 50 个会话。

这样的设计兼顾了真实产品能力和可用性：用户登录后可以跨刷新恢复，未登录或网络异常时也不会直接丢失对话。

### 5.2 数据结构

新增两张表：

- `chat_sessions`：保存会话标题、置顶状态、创建时间、更新时间。
- `chat_messages`：保存每条 user / assistant 消息，以及 assistant 当次回答的 `context_sources` 摘要。

关键边界是：不保存完整日历上下文和完整日记上下文，只保存来源摘要、缺失来源和可信度。

### 5.3 交互层

聊天页新增 260px 历史侧边栏，放在主导航和聊天内容之间。

历史栏支持：

- 新对话。
- 会话标题和更新时间。
- 搜索历史。
- 手动重命名。
- 删除确认。
- 置顶 / 取消置顶，最多 4 个。
- 本机 / 未同步状态提示。
- 移动端抽屉。

新对话采用“先进入空白页，等第一条消息发送时再创建会话”的方式，避免产生空会话。

### 5.4 模型上下文

每次请求模型时，只带当前会话最近 20 条消息。

这样可以让 Agent 理解本轮对话前文，又避免把全部历史无差别塞进上下文。日历和日记上下文仍然在发送时动态构建，保持和最新数据同步。

## 6. 关键取舍

一、先做会话级记忆，不做长期记忆。

长期记忆会涉及偏好提取、用户确认、记忆管理和删除权。第一版如果自动写入长期画像，容易越过隐私和产品边界。因此这次只做“当前会话可延续”。

二、保存来源摘要，不保存完整上下文。

用户需要知道回答参考了什么，但聊天记录不应该复制一份完整日历或日记数据。`context_sources` 只记录已参考、未参考和可信度。

三、云端优先，但必须可降级。

时间管理产品不能因为网络失败就丢掉用户刚刚说的话。localStorage fallback 让产品在异常情况下仍然可用，同时通过“本机”“未同步”告诉用户真实状态。

四、历史搜索做关键词，不做语义召回。

V1 只支持标题和消息内容搜索，范围限定在最近 50 个云端会话和最近 10 个本机会话。语义召回和跨会话上下文注入放到后续版本。

## 7. 工程实现

这次实现主要涉及：

- `components/Chat.tsx`：历史侧边栏、会话选择、发送流程、重命名、删除、置顶、移动端抽屉。
- `utils/chatHistoryService.ts`：统一封装 Supabase 与 localStorage 的聊天历史读写。
- `supabase/migrations/20260719_create_chat_history.sql`：新增聊天会话和消息表、索引、RLS policy。
- `supabase/migrations/20260719065000_tighten_chat_history_grants.sql`：收紧表权限，避免超出最小授权。

## 8. 验证结果

已验证：

- 未登录状态下发送消息会创建本地会话。
- user message 和 assistant message 都能保存。
- 刷新页面后会话和消息可以恢复。
- 多轮追问能引用上一轮上下文。
- 历史搜索能命中标题和消息内容。
- 搜索无结果显示“没有找到相关历史”。
- 手动重命名生效。
- 置顶生效，菜单变为“取消置顶”。
- 删除前有二次确认，删除后清理会话和消息。
- 移动端历史抽屉可打开，选择会话后关闭。
- Supabase 远端 migration 已同步。
- RLS、policy、index、grant 已核验。
- authenticated 角色可以创建会话、写入消息、更新会话、读取记录、删除记录。
- 构建和相关回归测试通过。

验证命令包括：

```bash
npx tsx tests/test-chat-history-service.ts
npx tsx tests/test-profile-avatar.ts
npx tsx tests/test-supabase-proxy.ts
npx tsx tests/test-deepseek-chat-mode.ts
npx tsx tests/test-chat-plain-text-prompt.ts
npx tsc --noEmit
npm run build
```

## 9. 最终效果

用户现在可以在聊天模块里看到历史会话，并在刷新后继续之前的对话。Agent 的回答不再只是一次性的临时响应，而是能在当前会话内承接用户刚刚讨论过的计划、状态和偏好表达。

这一步让 TimeTrack 的 Agent 能力从“上下文问答”推进到“可延续的个人时间对话”。

## 10. 作品集表达

可以这样讲：

> 我没有直接追求复杂的长期记忆，而是先补齐真实产品里最基础、也最容易被用户感知的会话级记忆。通过 Supabase + localStorage fallback 的数据层设计，用户刷新后仍能恢复对话；通过最近 20 条当前会话消息进入模型上下文，Agent 能理解刚刚讨论过的任务；通过来源摘要和 RLS 边界，我把隐私和可解释性也纳入了产品设计。

这体现的是三个能力：

- 产品判断：先解决用户真实会遇到的断点，而不是为了技术炫技直接做长期记忆。
- 工程交付：从数据库、RLS、fallback、UI 到验证闭环都完成落地。
- AI 产品边界：让 Agent 记住“当前会话”，但不自动保存长期偏好，不越权写日程。

## 11. 后续迭代

下一步可以继续做：

- 当前会话摘要，降低长会话 token 成本。
- 候选长期记忆，由用户确认后保存。
- 记忆管理页，支持查看、编辑、删除和关闭记忆。
- 更细的来源依据展开。
- Agent 工具调用，但所有写入动作都需要用户确认。
